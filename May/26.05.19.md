### 새로 배운 내용
#### 서블릿 (톰캣)

클라이언트와 소통하기 위해서 HTTP 응답(메소드, 헤더, 메세지 등)을 만들어야 했음

→ 개발자가 집중해야 하는 건 http 요청 < 비즈니스 로직 제작

→ 서블릿 메서드로 요청/응답 흐름을 제공

**서블릿 컨테이너 vs 빈 컨테이너**

- 빈 컨테이너 → 그릇 (객체를 담아둠)
- 서블릿 컨테이너 → 런타임 환경
- 빈 컨테이너와 서블릿 컨테이너 각각 하나가 같은 Dispatcher Servlet 를 품게됨
- 빈 컨테이너 매핑 → 서블릿 컨테이너 매핑

**Dispatcher Servlet (서블릿 컨테이너 내부)**
- Handler Mapping : 요청을 전달할 컨트롤러 찾기
- Handler Adapter : 컨트롤러 실행
- View Resolver : 반환할 뷰 찾기

→ 책임을 분할해줌

#### 스프링 MVC

MVC : 코드의 책임을 분리하는 패턴
- Model : 데이터, 비즈니스 로직
    - 조건 검사 및 데이터 관련 작업
- View : 사용자 화면
    - 결과물 관련된 것
    - Json, HTML, JSP …
    - 서버가 내용을 전달하면 프론트가 그림
- Controller : 요청 처리(Url 단위), 모델과 뷰 연결
    - 클라이언트 요청 해석 → 필요 모델 import

MVC 없이 웹 요청을 처리하면? 한 요청당 서블릿 하나를 만들어야돼서 기하급수적으로 늘어남

→ DispatcherServlet 을 이용하여 하나의 서블릿만 사용하고 해당 서블렛이 분배하게 함

소프트웨어를 만들면 디자인, 데이터, 통신 제어 영역으로 나눠서 개발하는 디자인 아키텍처

#### https

http 만으로도 충분했음 → 보안이 필요 없었음 (검색, 사전)

하지만 요즘은 본인 인증, 결제 등 보안이 중요해짐

1. 암호화 : 데이터를 읽을 수 없는 형태로 변환(평문 → 암호)
    1. 대칭키(비밀키) : 암호화 복호화를 1개의 키로
        1. XOR 연산
        2. 데이터 전달이 빠르지만 키 전달 어려움
    2. 비대칭키 : 암호화 공개키 / 복호화 개인키
        1. 연산이 많아서 CPU 자원 많이 듦
        2. 시간도 많이 듦
2. 인증
    1. 전자서명
        1. 비대칭키 이용 : 암호화 개인키(서명하는 사람) / 복호화 공개키 (암호와와 반대)(인증서)
    2. 인증서
        1. 내부 존재 항목
            1. **서버의 공개키(비대칭키로 서명된)**
            2. 인증기관 정보
            3. **인증기관의 전자서명(비밀키로 서명된)**
3. https 통신 순서
    1. 서버가 인증서를 만들기 위해 공개키/개인키를 만듦 → 비대칭키
    2. 인증기관을 통해 인증서 발급(서버 공개키/인증기관 비밀키) → 비대칭키
    3. 암호 통신 원할 때 클라이언트가 서버에 인증서 요청
    4. 인증기관 일치 여부 검증(서버의 공개키인 전자서명/인증기관 공개키 검증) → 비대칭키
    5. 클라이언트가 대칭키를 제작 by 인증서 안 서버의 공개키 이용 → 서버의 개인키로만 열 수 있음 → 비대칭키
    6. 서로 대칭된 키로 통신 → 대칭키
