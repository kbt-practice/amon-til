### 새로 배운 내용
#### 주제 1: 상태

**statefull**
- 상태를 기억함
- 복잡한 요청 처리 용이함 → 이미 그 상태에 정보가 존재하니까
- 상대적으로 느림
- 확장성 어려움

**stateless** → HTTP
- 상태를 기억하지 않음
- 서버 부담이 적음
- 처리 속도가 빠름 → 차지된 공간이 적으니까 널널함
- 복잡한 요청에 보낼 정보 많음
- 순서 관리 어려움

**connectionfull**
- 연결 맺고 유지
- 순서 보장되며 전송 실패 시 재전송 시도
- TCP, WebSocket

**connectionless** → HTTP
- 요청 후 바로 끊음
- UDP, HTTP

#### 주제 3: 페이지네이션
페이지네이션 (Pagination)
왜 필요한가?
전체 데이터를 한 번에 가져오면 DB 부하 폭증 + 응답 느림 → 데이터를 끊어서 가져오는 전략이 필요

1. 표현 방식

**페이지 기반 UI**

- 1, 2, 3... 번호를 클릭해 특정 구간으로 이동
- 특정 페이지로 바로 점프 가능
- 데이터 변화(추가/삭제) 시 번호 기준이 흔들려 중복/누락 발생 가능

**인피니티 스크롤**

- 스크롤 끝에 닿으면 다음 데이터를 자동으로 불러옴
- 특정 위치로 이동 불가 (뒤로가기 시 위치 초기화 문제)
- SNS, 피드처럼 순서보다 흐름이 중요한 UX에 적합


2. 구현 방식

**Offset 기반**

- LIMIT n OFFSET m → m개 건너뛰고 n개 가져옴
- 사용 이유 : 구현 단순, 페이지 번호 계산이 직관적
- 특정 페이지 바로 조회 가능
  - Offset이 커질수록 DB가 앞 데이터를 전부 읽고 버려야 해서 후반 페이지 조회 느림
  - 조회 도중 새 데이터 삽입 시 다음 페이지에 이전 데이터가 중복 등장 (데이터 밀림)

**Cursor(Key) 기반**

- 마지막으로 받은 데이터의 key(보통 id, created_at)를 기준으로 그 다음을 가져옴
- 사용 이유 : Offset 없이 특정 지점 이후만 조회 → 대용량에서도 빠름
- 인덱스 타고 바로 조회 → 데이터 양과 무관하게 속도 일정
- 조회 중 데이터 추가돼도 커서 이후만 보니까 중복/누락 없음
  - 특정 페이지로 점프 불가 (cursor가 없으면 접근 못 함)
  - 정렬 기준이 유니크해야 함 (동일 timestamp 다수면 cursor 모호)
