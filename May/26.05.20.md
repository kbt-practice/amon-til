### 새로 배운 내용
#### 주제 1: OOP
- 객체지향적 설계 : 데이터를 속성(필드)와 기능(메서드)로 묶어 프로그램을 설계하는 것
  - 누구에게 이 일을 맡길 것인가? → 데이터와 행동이 객체로 묶임
  - 객체 내부만 수정하면 돼서 영향이 다른 곳으로 가지 않음
- OOP 안 쓰면? → 규모가 커지면 유지보수 어려움 (많은 시간)

#### 주제 2: SOLID
- 유지보수성 + 확장성을 높이고자 만든 다섯가지 핵심 원칙

**SRP(Single Responsibility Principle) : 단일 책임 원칙**
- 클래스를 수정해야하는 이유는 단 한가지이다.
    
    → 수정 범위가 적어짐

**OCP : 개방폐쇄 원칙**
- 확장은 열어두고 기존 코드 수정은 닫아라
- 기능을 추가할 때에는 기존 코드를 건드리지 않는다
- 내가 개발하지 않은 코드가 존재함

**LSP : 리스코프 치환 원칙**
- 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 함

**ISP : 인터페이스 분리 법칙**
- 내가 안쓰는 거대한 인터페이스보다 쪼개진 인터페이스 쓰기

**DIP : 의존관계 역전의 원칙**
- 구체적인 것에 의존하지 말고 추상적인 것(구현되어 있지 않은 것)에 의존해라
- 상위모듈이 하위모듈에 의존하지 않고 둘 다 추상화에 의존
- 선언은 인터페이스(추상클래스)에 의존하고 구현을 함

#### 주제 3: JVM
**C언어와 자바의 차이점**
- C언어는 컴파일러를 통해 기계어가 나오는데 OS마다 종속되어 있음
- 자바는 바이트코드가 JVM으로 해석하여 다른 OS에서도 같은 환경에서 코드가 실행됨

**JVM**
1. 정의 : 자바 바이트 코드를 실행하는 (기계어로 실행하는) 가상 머신 엔진
2. 특징 : 단독 실행 불가
3. 작동 순서
    1. 자바 코드 작성 : .java 의 파일로 자바 소스 코드 작성
    2. 컴파일 : javac 명령어를 통해 .class 의 확장자를 가진 중간 코드(바이트 코드) 생성
        - 필요할 때 필요한 파일을 찾아서 메모리에 올림
        - 바이트 파일 : 메타데이터(클래스, 필드 등) / 상수 풀 / 메서드 / 속성
    3. 클래스 로딩(해석의 시작점_ : 위에서 생성된 코드를 메서드 영역에 로드함. 동적으로 이루어지는 지연 로딩(필요 시 작동)
        1. 로딩 : 외부 저장소에서 .class 파일을 찾아 읽고 메서드 영역에 올림
        2. 링킹 : 해당 클래스가 올바르게 정의됐는지 검증 → 준비 → 해결(연결)
            - Verifying(검증) : 읽어들인 클래스가 JVM 명세에 맞는지 확인
            - Preparing(할당) : 클래스가 필요로 하는 메모리 할당
            - Resloving(해결) : 심볼릭 → 다이렉트
        3. 초기화 : static 변수 정의된 값으로 초기화
4. 메모리 구성, 해석 : 인터프리터(실행 엔진)가 명령어 단위로 하나씩 읽고 해석하고 실행
5. JIT 컴파일 : 인터프리터에서 같은 메서드 반복 호출 시(count) 성능 보완을 위한 컴파일러 반복실행되는 핫스팟/핫코드를 번역없이 기존 기계어로 실행
6. 실행
7. 가비지 컬렉션(GC)
8. 영역

```
┌─────────────────────────────────────┐
│           Method Area               │  ← 모든 스레드 공유
│  (클래스 정보, static 변수, 상수)        │
├─────────────────────────────────────┤
│              Heap                   │  ← 모든 스레드 공유
│       (객체 인스턴스, 배열)              │
├──────────────┬──────────────────────┤
│   Stack      │   Stack   │  Stack   │  ← 스레드마다 개별
│  (Thread 1)  │ (Thread 2)│(Thread 3)│
├──────────────┴──────────────────────┤
│           PC Register               │  ← 스레드마다 개별
├─────────────────────────────────────┤
│       Native Method Stack           │  ← 스레드마다 개별
└─────────────────────────────────────┘
```
a. 메서드 영역
- JVM이 시작될 때 생성되는 메모리 공간
- 바이트 코드를 올릴 때 초기화 되는 대상
- 모든 스레드가 공유함
- 프로그램 종료 시까지 저장
- 런타임 상수 풀, 필드/메서드 데이터, 메서드 코드, 생성자 코드

b. 힙 영역
   - 모든 객체의 인스턴스와 배열(실체) JVM이 실행 중 동적으로 저장
   - 모든 스레드가 공유함
   - new 키워드로 생성된 참조 타입 객체 저장
       - 클래스, 인스턴스, 이넘, 배열
   - 참조되지 않을때까지 유지되며, GC에 의해 제거됨

c.스택 영역
   - 객체의 주소값 저장
   - 스레드가 시작되고 삭제될 때 생명주기 동일 (스레드 단위 생성)
   - 메서드를 실행할 때 사용 (임시 방)
   - 기본형 변수, 메서드 실행 정보
   - 크기가 작아 Stack Overflow 오류 자주 남
   - 후입선출 구조로 동작

d. PC 레지스터
   - 스레드의 생명주기 동일
   - JVM 명령어 주소 저장
   - 실행 흐름 기록 장치
