### 새로 배운 내용
#### 주제 1: 데이터베이스

**필요한 이유**

- 원래는 파일 시스템으로 관리했으나,
    - 데이터 기록 중복 → 중앙 집중 관리
    - 파일 커지면 찾기 힘듦 → 효율적 검색 (인덱스 등)
    - 보안/권한 부족 → 높은 보안성 (사용자/DB별 권한 제어)
    - 동시 수정 불가 → 동시성 제어(트랜잭션/락)
    - 데이터 잘못 적힘 → 무결성 제약
    - 다른 OS 일치하지 않음 → 호환성 제공
      - 관계형 데이터베이스 : 데이터를 표 단위로 관리하고, 관계를 표현
          - 사용법
              - 엔티티 : 데이터베이스에 넣고자 하는 대상 (학생)
              - 속성 : 대상이 가진 정보 (이름)
              - 관계 : 테이블끼리의 연결 (학생 - 강의)
                - 종류
                    - DDL : 데이터 정의어 - 테이블 관련 명령어
                    - DML : 데이터 조작어 - 데이터 조작 명령어
                    - DCL : 데이터 제어어 - 관리자 관련 명령어
                      - **왜 나눠져 있는가?**
      DB 작업의 성격이 완전히 다름 → 구조 정의 / 데이터 조작 / 권한 관리를 분리해야 실수 방지 + 권한 체계 명확
                      - 한눈에 비교
                      
                      | 분류 | 대상 | 롤백 | 주체 |
                      |---|---|----|---|
                      | DDL | 구조 (테이블) | X  | 개발자/DBA |
                      | DML | 데이터 (행) | O  | 개발자/앱 |
                      | DCL | 권한/트랜잭션 | —  | DBA |

#### 주제 2: ERD
- 구현 순서
    1. 개념적 모델링 : 피터 챈 표기법. 다이어그램
    2. 논리적 모델링 : 까마귀 발 표기법. 테이블의 관계
    3. DB로 구현
- ERD 표기법
    - Crow’s Foot Notation (까마귀 발 표기법)
        - 테이블 간 최대관계와 최소 관계를 직관적으로 표현
    - Peter Chen Notation (피터 챈 표기법)
        - 속성과 관계를 세밀하게 표현
    - Barker Notation (바커 표기법)
        - 엔티티 간의 관계를 더 명확하고 구조적으로 표현

- 식별(실선) : 부모-자식 관계여서 구분할 때 부모의 id를 사용
- 비식별(점선) : 본인만의 id가 필요함. 독립적인 테이블.
    - 더 선호 → why?
