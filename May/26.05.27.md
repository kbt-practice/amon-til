#### 주제 1: HTTP 자격증명헤더

보호되어야 하는 리소스(내 정보, 결제 정보 등) + 나를 증명해야함

but stateless 여서 매번 정보를 보내야함 → 헤더를 통해 인증하는 과정을 거쳐 불편함을 해소

**헤더 종류**

WWW-Authenticate

- 서버(response header에) → 클라이언트

Authorization

- 클라이언트 → 서버
- RFC ← HTTP 의 약속을 정하는 곳 : 인증정보는 앞으로 헤더에 담자는 약속

**인증 헤더 방식**

1. Basic Authentication
    1. 기본 인증. 아이디 + 패스워드
    2. Base 64 인코딩 된 평문
    3. HTTPS TLS 방식의 암호화된 통신에서만 사용
    4. 단) 서버가 확인해야되기 때문에 서버 자원에 부담됨
    5. 장) 구현이 간단함
2. Digest Authentication
    1. 기본 인증 보안이 취약한 이유 : 항상 비밀번호를 헤더에 담아 보냄 → 탈취위험
    2. Nonce : 서버 → 클라이언트 일회성 난수
    3. Realm : 서버가 관리하는 보안 공간의 범위 표기
    4. 클라이언트 요청 → 서버가 일회성 번호 줌 → 클라이언트가 비밀번호 + 해시 = 결과값 전송
3. Bearer Token
    1. 기존 방식의 문제점 : 매 요청마다 서버가 체크
    2. 로그인 후 토큰 발급 → 이 토큰으로 인증, 비밀번호 조회 1회만
    3. 장) 비밀번호 보호 및 범용성 좋음. CSRF
    4. 단) 탈취 쉬움 → 유효기간 짧게 설정
    5. 단) 보관이 까다로움
        1. 스토리지 : XSS 스크립트 해킹에 토큰 탈취 가능
        2. 쿠키 : CSRF 공격에 취약 → 방어 코딩 설정
    6. 토큰은 의미없는 무작위 문자열임 → 서버에 보관해야됨 → 서버 부담
    7. 그래서) 토큰에 유저 정보가 적혀있고 정보가 위조가 안됐다고 증명할 수 있으면 서버가 자원을 안써도 됨 → JWT

#### 주제 2: JWT

정보를 안전하게 전달하기 위한 웹 표준 개방형 규격

**구조**

```json
eyJhbGciOiJIUzI1NiIsIn... . eyJzdWIiOiIxMjM0NTY3O... . SflKxwRJSMeKKF2QT4fwpMe...
      [1. Header]        .       [2. Payload]       .       [3. Signature]
```

from jwt.io

1. 헤더 : 토큰의 타입과 알고리즘 종류
2. 페이로드 : JWT 토큰에 담을 정보들이 있음
    
    ex) 이메일, 유저이름, 발행일, 만료일 등 → 인코딩이라 디코딩 가능해서 노출 위험O
    
    - 발행일 만료일은 유닉스 타임스탬프 형식으로 발생
3. 서명 **:** 토큰이 위조되었는지 검증
    - 인코딩 된 헤더 + 인코딩 된 페이로드 + 서버만 알고 있는 비밀 키 값
    - JWT의 내용을 수정할 수 있음 → 서명이 달라짐 → 서명 검증에서 탈락하기 때문에 위조를 막을 수 있음

**장점**

- 완전한 무상태성 + 확장성 : 기존에는 서버 검증 로직 필요 → 이미 JWT 토큰에 저장되어 있어 해석 가능하고, 암호화된 서명만 검증하면 됨
- DB 조회 적게 함
    - 병목 지점 : DB 조회/쓰기. 가장 느린 쓰기 장치(HDD)에 기록됨
    - 인증 인가 필요 시 JWT 말고 다른 인증들은 DB 조회가 필요함

**단점**

- 무효화 어려움 → 스토리지 또는 쿠키
    - 해커가 탈취하면 만료시켜야되는데 못함 → 액세스 짧게/리프레시로 재발급
- 네트워크 대역폭 낭비
    - 토큰은 내부 내용이 많을 수록 길이가 길어짐 (인코딩/디코딩 기반이라)
    - 서버비가 증가함

**보안과 편의성의 딜레마** : Access/Refresh

- 15분마다 만료 → 15분마다 로그인 → 유저 불편함, 보안 좋음
- 2일 만료 → 탈취되어도 2일 동안 해결 불가 → 유저 편함, 보안 별로
- 리프레시 토큰 탈취 해결 방법 : RTR
    - 리프레시 1회성 사용
    - 리프레시 한번 사용 시 검증 후 새로운 액세스와 리프레시 토큰 발급
    - 탈취 됐을 때 기존 리프레시 토큰이 검증이 안되니까 거부

#### 주제 3: 레이어드 아키텍처
- 책임 분리해, 각 계층이 정해진 역할만 수행하도록 구성하는 구조
- 코드 간 관심사 분리

- Controller : 외부 입출력 관련 처리
    - 레이어별로 필요한 데이터가 다르기 때문에 전달할 데이터의 구조를 정의
    - DTO(전달할 데이터 구조) 정의
- Service : 비즈니스 로직 처리
    - 요청이 어떤 의미를 가지는지 해석하고 여러 작업을 어떤 순서로 수행할지를 결정
- Repository : 데이터베이스 CRUD 처리
    - 데이터베이스의 규칙을 객체 지향 코드에서 사용하기 위해 RDB 테이블 구조와 규칙을 객체화
    - ENTITY 테이블과 키를 정의하고 컬럼을 매핑
- Middleware : 필터와 인터셉터가 중간에서 먼저 확인
    - 클라이언트가 보낸 데이터에 필요한 데이터가 다 있나?
    - 매번 컨트롤러에서 확인해야할까?
    - 계층간 전달되는 데이터들은 적절한가?
