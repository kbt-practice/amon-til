  **클라이언트-서버 구조**
- 클라이언트 : 통신의 시작점. 서비스를 요청하는 단말기 또는 소프트웨어(유저 에이전트)
    - 사용자 인터페이스 구현
    - 크롬, 사파리, 엣지, CLI 등
- 서버 : 클라이언트에게 서비스나 자원을 제공하는 컴퓨터 시스템이나 프로그램
    - 비즈니스 로직 처리
    - 항상 포트를 열어두고 귀기울임
#### HTTP
- 하이퍼텍스트를 전송할 때의 전송 방식 규정
- 만약 없다면? 여러 방식의 하이퍼텍스트 전송 방식 → 꼬임
- OS를 따지지 않음
- 클라이언트가 URL로 특정 웹 주소에 요청/응답 

##### Message
1. 클라이언트와 서버가 주고받는 글
    1. Request : 클 → 서
    2. Response : 서 → 클
2. 구성
    1. 시작 줄(start line) : 상태 정보
    2. 헤더(header) : General/Req, Res/ Entity
    3. 빈 줄(empty line) : 헤더랑 본문 구분
    4. 바디(body, payload)  : 실제 내용
##### Method
1. 종류
    1. GET(조회)
    2. POST(생성)
    3. PUT(전체 수정)
    4. PATCH(일부 수정)
    5. DELETE(삭제)
2. 메서드를 나누는 이유
    1. 조회는 캐싱된 데이터 우선 조회하기 때문에 나누는 게 병목현상이 덜 생김
    2. 수정하고 삭제할 때 인가(보안) 고려
    3. 결론적 : 전체 내용을 훑어보지 않더라도 분류하고 처리하기 위해서
3. put/patch 구별법
    1. 팀 규약에 따라 결정
    2. 객체 기준으로 봄
##### Status Code

| 번호대 | 설명 |
| --- | --- |
| 1xx | 정보 메시지 (Informational) |
| 2xx | 성공 (Successful) |
| 3xx | 리다이렉션 (Redirection) |
| 4xx | 클라이언트 오류 (Client Error) |
| 5xx | 서버 오류 (Server Error) |
- 500에러 : 어떤 이유로 서버에서 실패했는지 응답하지 않음. 그냥 서버 에러로 통일해서 응답함.

##### URL
   1. 웹 리소스를 접근하기 위한 주소
   2. 스키마 Domain Port 경로 매개변수 로 이루어짐
   3. 포트는 nmap 명령어로 볼 수 있음
   4. 종류
       1. Query String : 웹페이지에 대한 추가 정보 제공
       2. Path Variable : 경로 자체에 변수 값 포함하여 식별함

#### 스프링

  등장 이유 : EJB 등의 어려움 → 객체지향스럽게 사용하기 위해서

  **특징**

1. IoC : 객체 관리를 컨테이너가 담당
    1. new → 설정/어노테이션 DI(의존성 주입)
    2. DI :  의존성 주입 (@Autowired)
        1. 필드 주입 : private
        2. 생성자 주입 : 생성자에
        3. 세터 주입 : public setter에
2. 경량 & 유연성 : 순수 자바로 개발 가능. 모듈 선택
3. AOP : 공통 관심사와 핵심 관심사 분리
4. 테스트 용이성 : 객체가 컨테이너에 관리되어 쉬워진 테스트

  **bean 등록 방법**

1. @Componetn @Service @Repository
2. Bean을 사용한 자바 설정
3. XML 설정 파일에 <bean> 태그 작성
