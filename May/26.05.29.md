#### 주제 1: 동시성 제어
- 데이터의 일관성과 무결성을 유지하는 기술

Serial Schedule : 트랜잭션이 서로 겹치지 않고 한번에 하나씩 실행되는 경우 → 일반적임

- 일관성 보장, 동시성 낮음

Non-Serial Schedule : 트랜잭션이 겹치게 처리됨

- 일관성 보장 낮음, 동시성 높음

confict : 아래 세가지 조건을 모두 만족 시 충돌

- 서로 다른 트랜잭션 소속
- 같은 데이터에 접근
- 최소 하나 Write

→ 판단 기준 : 직렬화 가능한가? (Precedence Graph로 판단)
→ 해결 방식 : Lock / MVCC / 타임스탬프

동시성 제어 방식 : **락킹** → 멈춰!

1. 비관적 락
    1. 충돌이 일어날 가능성을 가정
    2. 데이터 사용하기 전 락을 걸어 다른 접근을 막음
    3. RDBMS에서 주로 사용
    4. 작업할 자원에 접근을 제어함
    5. 높은 무결성, 데이터 손상 방지, 신뢰성 제공, 충돌 방지
    6. 성능 저하 및 데드락, 트랜잭션 대기 시간, 자원 사용 효율성 낮음, 과도한 락으로 인한 병목
        1. 데드락? 여러 트랜잭션이 서로가 가진 락을 기다리며 순환 대기 상태에 빠질 때 발생
        2. 데드락 방지
            - 트랜잭션 자주 COMMIT
            - 정해진 순서로 테이블에 접근
            - 읽기 잠금 획득(SELECT ~ FOR UPDATE)의 사용을 지양
2. 낙관적 락
    1. 충돌이 일어나지 않을거라 가정
    2. 데이터 베이스 자원에 대한 동시 접근 방지
    3. NoSQL에서 주로 사용
    4. 높은 동시성, 성능 향상, 분산 시스템 적합, 빠른 읽기 속도
    5. 무결성 낮음, 복잡한 예외 처리, 추가 구현 비용, 충돌 시 재시도 비용 증가

#### 주제 2: OAuth
- 인증과 인가를 분리함으로 외부 서비스에 정보를 제공함
    - 인증은 유저가 직접하고 인가는 앱이 받음
- 동작 방식
    1. 사용자 로그인 요청
    2. 클라이언트가 인증 요청을 인증 서버로 전송
    3. 로그인 페이지 제공
    4. 사용자의 ID/PW 입력 및 제출
    5. Authorization Code 발급
    6. 클라이언트로의 리다이렉션
    7. Authorization Code와 Access Token 교환
    8. Access Token (및 필요시 Refresh Token) 발급
    9. 로그인 성공
    10. Access Token으로 리소스 서버에 리소스 요청
    11. 리소스 서버의 토큰 검증
    12. 리소스 제공
    13. 클라이언트에서 데이터 활용
- Autorization Code를 사용자에게 바로 주는 이유
    - HTTP Stateless → 인증 요청 후 받는 것인데, 저장이 안되면 인가 시 정보를 알 수 없음
    - 보안성 → Access Token을 브라우저 URL에 노출하면 위험
    - 사용자가 oauth 와 통신하며 인증하고, 서버에게 인가 권한을 주는 것이라서
    - server의 secert을 모르기 때문에 탈취당해도 괜찮음

- PKCE : 로그인마다 임시 비밀번호를 만들어 사용하는 인증 방법
    - 웹앱에서 직접 액세스 토큰을 교환함
- 사용 이유
    - 인증과 인가를 분리해서 쉽게 사용하기 위해서
    - 서버의 부담을 줄임
