### 새로 배운 내용
#### 주제 1: 객체지향
- 자바 추가 공부 (교재)
    1. 다른 패키지의 같은 이름을 import 해올 때에는 하나는 import 문을 쓰고 하나는 경로 명시를 해서 가져옴
    2. static 키워드는 클래스 단위로 공유함 → 전역변수로 생각. 접근 제어자에 따라 다른 클래스에서도 사용 가능
        1. 메인, 모든 인스턴스가 접근하는 변수, 유틸리티
        2. 남용 시 객체지향 설계 위반, 메모리 누수 등의 문제 발생
    3. 문자열 값을 비교할 때는 (==)가 아닌 equals() 메서드를 사용해야함
    4. `public static void main(String[] args)` : 공개 바로실행 값변환없음 이름(외부입력값)
        1. 외부입력값은 비워도 되지만 시그니처이기 때문에 맞춰주기
        2. 접근 제어자
            1. 전체 접근 : public
            2. 자식까지 : protected
            3. 같은 패키지만 : default
            4. 같은 클래스만 : private
- 객체지향

  객체 = 부품 이라고 생각하면 편하다

  재사용성, 유지보수성, 대규모 협업 가능

  특징 : 캡슐화, 상속, 다형성, 추상화

- 상속
    1. 클래스 관계가 분류라면 상속 ex) 무기 - 검, 활, 도끼
    2. 개념이 나뉘어 분류가 필요할 때 상속 ex) 아이스아메리카노, 아메리카노 → 아메리카노 + 얼음
- 다형성

  오버로딩 → 생성자에 사용

    - 코드의 일관성과 가독성을 높임
    - 같은 메서드 이름, 다른 매개변수 → 다른 동작
    - return 타입 유지 필요 없음

  오버라이딩

    - 상속, 인터페이스에서 사용. 자식의 개성대로 동작하게 만드는 객체지향의 특징
    - 같은 메서드 이름, 같은 매개변수 → 동작 추가 or 달라짐
    - return 타입 유지
- 캐스팅

  업캐스팅 : 부모 타입의 자식객체 생성

  다운캐스팅 : 업캐스팅 한 것을 다시 자식으로 돌려 자식 클래스 기능 사용

- 캡슐화
    - 클래스 변수를 private로 선언함
    - 왜 필요함
        - 데이터 보호, 유효 검사를 위해 사용
        - 내부 구현 은닉을 위해 사용
    - getter(가져옴) setter(설정함)
- 인터페이스
    1. 유연성
    2. 병렬 개발 : 설계자가 인터페이스를 만들고 구현자가 알아서 각각 기능을 구현함

  상속과 인터페이스의 차이

  | 분류 | 상속 | 인터페이스 |
      | --- | --- | --- |
  | 키워드 | Is-A | Can-Do |
  | 상속 | 단일 상속만 가능 | 다중 구현 가능 |
  | 관계 | 수직적, 계층적 구조 | 수평적, 기능적 확장 |
    - 공통점은 상속으로 묶고, 차이점은 인터페이스로 분리한다.
    - 인터페이스는 확장성과 유연성을 만듦
      - ex) 하늘을 나는 자동차가 생기면 추후에 fly 인터페이스만 추가하면 됨

#### 주제 2: 제네릭
- 제네릭

    - 어떤 타입이 들어올지 모르기 때문에 제네릭이란 걸 만듦 → 잘못된 타입 시 에러 발생, 타입 변환 생략

    - 사용하는 경우
      1. 공통의 API 응답 규격을 만들 때
      2. 컬렉션 프레임워크
          - 여러개의 데이터를 쉽고 효과적으로 처리할 수 있도록 자바에서 미리 만든 표준화된 자료구조 세트 → 인터페이스

         list
          - 순서가 있는 유형의 데이터 집합
    
         set
          - 중복없는 집합
          - 순서를 보장 X, 중복 허용 O
    
         map
          - 키와 값이 한 쌍인 자료 구조
          - 키:값 → 키를 기반으로 검색. 키는 중복 X, 값은 중복 O
    
    - 제한된 타입(Bounded Type)
    
        - 과도한 타입 자유는 위험함 ex) 숫자 계산인데 문자열 넣으면 오류 발생
        - <T extends 부모클래스> : T에 들어올 수 있는 클래스는 반드시 부모 클래스이거나 자식 클래스여야 함
        - 자바는 안전을 추구하기 때문에 보통 제한된 타입의 제네릭을 많이 씀
        - 와일드카드
- Thread

  프로그램(프로세스) 내 실행 흐름

  싱글 스레드

    - 한 명이 모든 일을 다 함

  멀티 스레드

    - 여러명이 일을 함께 함

  **왜 사용하는가?** 멈추지 않고 동시에 수행하기 위함

    - 응답성 유지 : 영상 다운 받으면서 동시에 채팅 전송
    - 백그라운드 처리 : 화면에는 보이지 않지만 여러가지 일을 동시에 처리함
    - 자원 효율 극대화 : CPU(코어) 가 여러개면 여러 일을 수행시킬 수 있음
    - 대용량 데이터 배치 작업 : 병렬 실행하여 다른 일도 할 수 있게 함

  주의사항

    1. 예외처리 해주기
    2. 가시성 문제 : 접근 순서에 따라 다르게 봄
    3. 순서가 지켜지지 않을 수 있음

  Runnable : 새로운 실행 흐름 작업의 정의서(인터페이스)

  스레드는 클래스임 → 상속은 하나만 가능

  → 만약에 상속 받아야 할 대상이 있다면 스레드 자체 상속 불가

  → 스레드 안에서 필요한 함수를 포함시키기 위해서 인터페이스 설정(러너블)을 해줌

  스레드는 일꾼 - 실제 흐름 및 제어 / 러너블은 - 어떤 걸 하는지 알려줌

  스레드 구현 방법

    1. 클래스
    2. 익명 클래스
    3. 람다식
    4. ExecutorService로 러너블 단위로 작업함 (실무)

  스레드 안에 러너블을 감싼 클래스를 감싸서 생성 → 포함 → 다른 클래스 상속 받을 때

  스레드를 부모클래스로 해서 상속 관계 설정 → 상속 → 다른 클래스 상속받지 않을 때

- ExcutorService

  new Thread의 위험성

    - 스레드는 일반적으로 직접 생성하지 않음
        1. 생성 비용이 높음
        2. 리소스 고갈 문제(OOM)
        3. 관리가 힘듦 → 명시적 종료 선어 필수

  ExcutoreService 라는 업체를 이용해서 스레드를 사용함

    - 스레드 풀을 미리 만들어둠 → 순서에 따라 배치하고 사용하게 함 → 생성/삭제 줄어듦
- Synchronized

  여러 스레드가 동시에 공유 자원에 접근하지 못하도록 접근을 스레드 하나로 제한

  스레드 별로는 스택 공간을 각각 차지하고 공용공간은 힙을 사용함

  → 임계영역 정하기

- Lock

  Synchronized는 대기 스레드 중 누가 먼저 왔는지 확인이 어려움 (교착상태에 빠질 수 있음)

  Lock은 누가 먼저 왔는지 알 수 있어 먼저 처리할 수 있음. 대기하다 오래걸리면 탈출도 가능
