### 새로 배운 내용
#### 주제 1: NoSQL
- Not Only SQL : SQL만 사용하지 않음
- 필요한 이유 : RDB의 한계
    - 정규화된 테이블 구조 : 일관성을 보장하지만 유연하지 못함
    - 트랜잭션 → 원자성을 보장하기 때문에 동시 처리성이 낮음
        - 원자성 : 전부 성공 또는 전부 취소
    - 디스크 기반 → 확장 어려움
- 대용량 데이터베이스 적절
- 컬럼 단위 데이터 → 읽기 성능 좋음
- 설정 복잡

**InMemory**

- 데이터를 RAM에 올려두고 처리하는 방식
- 메모리(RAM) 특징
    - 데이터 날아감 (휘발성)
    - 속도 빠름
    - 용량 적음
    - 가성비 안 좋음
- 메모리를 쓰기 좋은 경우
    - 데이터가 휘발되도 크게 지장이 없음
    - 자주 접근되어서 속도에 많이 예민
    - 저장되는 데이터가 복잡하지 않고, 작은 데이터
    - 적은 용량으로도 서비스가 가능한

**키-값 저장소**

- Redis, AEROSPIKE
- 동작 ****방식
    - 쓰기 : 키를 해싱하여 데이터 저장 위치 결정
    - 읽기 : 키를 입력받아 데이터를 검색
    - 샤딩 : 해시 함수를 사용해 데이터를 여러 노드에 분산 저장
- 장점
    - 매우 빠른 읽기/쓰기 속도 → key-value만 있음
        - 인메모리 구조
        - 해시 테이블 기반
    - 구조 단순, 수평 확장 용이함
- 단점
    - 데이터 구조가 단순해서, 복잡한 쿼리나 검색이 어려움
    - value 내부 구조를 DB가 이해하지 못함
- 사용처
    - 자주 접근하는 데이터라 속도가 중요하거나 없어져도 크게 문제가 되지 않는 데이터
        - 세션, 토큰, 설정 등등
        - 많이 써야하고 많이 읽혀지는데 속도 중요한 곳

**Documents Store/Document Oriented Databases (문서 저장소)**

- MongoDB, CouchDB
- JSON, XML과 같은 문서 형식을 사용하여 데이터를 저장
- 동작 방식
    - 쓰기 : 고유 ID로 문서를 저장
    - 읽기 : ID 또는 특정 조건(필드 값)을 기준으로 문서를 검색
    - 인덱스 : 특정 필드에 대해 인덱스를 생성하여 조회 속도를 최적화
- 장점
    - 구조가 자유로워 스키마 변경이 유연
    - 문서 단위로 조회, 수정 가능
    - 쿼리 검색 유연
- 단점
    - JOIN이 어려움 → 복잡한 관계 데이터는 비효율적
    - 데이터 중복 허용
    - 데이터 간 관계성이 약함

주사용처

- 콘텐츠 관리, 블로그, 게시판, 카탈로그, 채팅 시스템
- 속성의 수정이 많거나, 관계성이 낮은 데이터

**Column-Oriented Store/Column-Family Store (컬럼 기반 저장소)**

- 컬럼 단위로 데이터를 저장하고 관리
    - 이름끼리 저장, 급여끼리 저장
- 대용량의 데이터를 빠르게 조회하고 분석할 수 있음
- Cassandra, HBASE
- 동작 방식
    - 쓰기 : RowKey와 Column 기준으로 데이터를 저장
    - 읽기 : 특정 RowKey 또는 Column 기반으로 데이터 검색
    - 샤딩 : RowKey를 기준으로 데이터 분산
- 장점
    - 대용량 데이터, 빅데이터 처리 적합
    - 컬럼 단위 조회 최적화 → 읽기 성능 좋음
- 단점
    - 설계가 복잡함. 스키마 잘못 설계하면 비효율
    - 단순 조회/저장 외 복잡 쿼리는 어려움
- 주 사용처
    - 로그 수집, 분석, 시계열 데이터, 추천 시스템
    - 속성의 수정이 많거나, 관계성이 낮은 데이터

**Graph Store/Graph Database (그래프 데이터베이스)**

- 데이터를 노드와 엣지로 표현하여, 객체 간의 관계를 중심으로 데이터를 저장하고 조회
- Neo4j
- 동작 방식
    - 쓰기 : 데이터는 노드와 간선으로 저장
    - 읽기 : 노드와 관계를 탐색
    - 인덱스 : 노드 및 간선에 대한 인덱스를 생성하여 탐색 속도 향상
- 장점
    - 관계 기반 조회 최적화
    - 소셜 네트워크, 추천 시스템에서 효율적
- 단점
    - 트랜잭션/대용량 처리 복잡
    - 수평 확장이 어려움(관계를 유지해야함)
- 주 사용처
    - 소셜 그래프, 추천 엔진, 지식 그래프

#### 주제 2: 데이터베이스의 확장
- 데이터베이스를 물리적 또는 논리적으로 분산시키는 기술
- 부하 분산과 시스템의 확장성 향상을 목표로 함
- 처리 한계 해소 - 시스템의 무중단 운영과 효율적인 자원 활용
- 데이터베이스 확장을 통해 얻는 이점
    - 데이터 관리와 접근성 개선
    - 시스템 장애에 대한 내성 갖춤

**파티셔닝**

- 기존 테이블을 파티션으로 나눔
- 수직 파티셔닝 : 열 기준 분리
    - 자주 사용 / 사용 안하는 열로 분리
    - 조회가 잦은 데이터에 대한 I/O 감소, 테이블 잠금, 캐싱 효율 증가
    - 조인 필요도가 늘어서 쿼리 복잡도 증가, 2개 테이블 동시 적용 트랜잭션 시 관리 어려움
- 수평 파티셔닝(샤딩) : 행 단위로 기준 분리
    - 테이블 구조는 동일하지만, 저장되는 행위 범위가 서로 다름
    - 특정 범위 데이터만 조회하도록 하여 스캔 비용 감소
    - 많이 작성되는 경우 쓰기 성능 증가, 특정 파티션 별로 처리하기 좋음, 아카이빙/백업 쉬움
    - 인덱스 관리 힘듬, 데이터 불균형, 파티션간 쿼리 검색이 어려움, 트랜잭션 어려움

**복제 (Replicaion)**

- 하나의 데이터베이스에 저장된 데이터를 여러 노드로 복사하여 유지하는 기술
- 고가용성 제공, 읽기 부하 분산, 장애 발생 시 빠른 복구 가능
- 데이터 동기화로 인한 문제 발생, 스토리지 비용 증가, 대역폭 요구 사항 증가

**샤딩 (Sharding)**

- 데이터를 Shard라는 작은 청크로 분할하여 여러 데이터베이스 서버에 저장하는 것
- 수평 확장 가능, 응답 시간 개선, 전체 서비스 중단 방지
- 데이터 핫스팟 발생, 운영 복잡성 증가, 인프라 비용 증가, 애플리케이션 복잡성 증가
- 범위 기반 샤딩, 해시 샤딩, 디렉터리 샤딩, 지리적 샤딩

**Sharding vs Horizontal Partitioning**

- 수평 분할은 **같은** 데이터베이스 내에서 하나의 큰 테이블을 쪼개 분산 저장
- 샤딩은 하나의 **큰** 테이블을 쪼개 다른 데이터베이스에 분산 저장하는 기법

### 주제 3: 서버 캐시
- 웹 서버가 데이터를 빠르게 제공하기 위해 자주 사용되는 정보를 임시로 저장하는 장소
- 웹 사이트의 속도를 높이고 서버 부하를 줄임
- 많은 클라이언트가 똑같은 데이터를 요청을 하는데 일일이 데이터베이스에 요청을 한다면 너무 비효율적

**로컬 캐시**

- 각 서버 인스턴스에 독립적으로 존재하며, 서버의 리소스를 사용하여 데이터를 캐싱함
- 데이터 접근 속도 빠름, 추가적인 네트워크 지연 없음
- 스케일 아웃(수평 확장, 즉 서버 대수 늘리기) 시 일관성 유지 힘듬, 각 서버 캐시가 독립적으로 작동

**글로벌 캐시**

- 여러 서버 인스턴스 간에 공유되며, 네트워크를 통해 접근 가능한 별도의 캐시 서버에서 데이터를 관리함
- 데이터 일관성과 공유, 동일한 캐시 데이터 사용
- 네트워크 지연 가능성, 로컬에 비해 상대적으로 느림

**캐시 사용 예시**

- DB: DB 엔진 메모리에 데이터 페이지 및 쿼리 결과 저장
- 스토리지: 정적 리소스를 CDN에 저장
- CDN : 전 세계 여러 지역에 둔 서버(엣지 서버)에 콘텐츠 사본을 미리 캐싱해두고, 사용자가 요청하면 가장 가까운 서버에서 전송하는 시스템
- 브라우저: 로컬 브라우저에 HTML, CSS, JS 등을 저장
- 프록시: HTTP 응답 캐싱, WAS 전에 정적 응답 캐싱
