#### 주제 1: JPA 값

기본 값 타입 `@Entity`

단일 값으로만 의미를 가지는 데이터를 엔티티의 필드로 직접 정의해 테이블 컬럼에 저장하는 방식

- 기본형 : int, long, duble …
- rapper 클래스 : Integer, Long, Double …
    - ID값으로 많이 사용되는 이유 : 기본형을 객체로 감싼 타입. null을 가질 수 있음 → 아직 영속화되지 않은 상태(미저장) 알 수 있음 → 객체이기 때문
- 문자열 타입 : 자바의 기본 자료형으로 엔티티의 필드로 선언
- 생명 주기 엔티티에 종속, 값이 바뀌면 교체됨

**임베디드 타입** `@Embeddable`

테이블은 그대로 엔티티 구조만 더 객체 지향적으로

2개 이상의 값을 하나의 객체로 묶어 엔티티에 포함함

별도 PK, FK 없음

현업에서 많이 쓰진 않음

**컬렉션 값 타입** `@CollectionTable`

엔티티에 직접 연관되지 않는 여러개의 값을 리스트나 set 형태(컬렉션)로 저장

- 왜? 객체 지향적으로 보려고 부모객체와 묶어서 쓰려고

별도의 컬렉션 테이블이 생성되고, 소유 엔티티 PK를 FK로 가짐

#### 주제 2: JPQL

JPA만으로 조건, 조회 시 : 엔티티 → 연관 엔티티 순차 조회

엔티티 기반으로 데이터 Get하려면 레포지토리 자바 코드에서 조건, 정렬, 검색 등 진행

= 조회 → 초기화 → 정렬

만약 게시글이 너무 많으면? 메모리 부담 + 불필요 → JPQL 사용

JPQL : 쿼리 한 번으로 정렬+조건 처리 / Lazy X / 데이터 처리 성능 유리

**특징**

- 엔티티 중심의 쿼리 언어
    - SQL은 테이블 단위, JPQL은 객체 단위
- 엔티티 모델을 기준으로 DB에 맞게 SQL문 작성됨 → DB 문법 종속 X

**사용법**

```sql
select 별칭 // 별칭 필수
from 테이블명 별칭 // 엔티티 기반으로 쿼리 작성
where 조건식
group by 그룹핑기준
having 그룹핑조건
order by 정렬 기준
```

- 엔티티 매니저를 통해 쿼리 생성
    - createQuery, getResultList, getSingleResult, executeUpdate
- 경로 표현식
    - 묵시적 조인 : join 키워드 없이 경로만 표현해서 항상 내부 join
    - 명시적 조인 : 복잡할수록 성능이 떨어짐 → join 키워드로 선언 → 권장
- N+1 문제 : 지연로딩 시 자주 발생하는 문제
    - user 조회 시 post 까지 한 번 더 조회 → N개 조회하려는데 1번 더 조회함

프로젝션 : 필요한 부분만 조회하는 것

- 스칼라 : 오브젝트 or 튜플로 단일 컬럼들 조회
- DTO : DTO 객체에 있는 것들만
- Entity : 엔티티 자체를 조회

#### 주제 3: Spring Data JPA

엔티티 매니저 직접 호출 거의 X

인터페이스만으로 레포지토리를 구현 → 개발 생산성과 유지 보수 효율을 높임

순수 JPA는 엔티티 매니저를 관리해야함

근데 CRUD 레포지토리는 거의 반복됨 → JpaRepository 상속받아 CRUD 자동 제공

내부적으로 순수 JPA 상속받음

1. 기본 사용법

    ```sql
    @Repository
    public interface 레포지토리명 extends JpaRepository<엔티티, ID> {
        ...
    }
    ```

2. Query Method : 메서드 이름을 기반으로 JPQL 쿼리 자동 생성
    - ex) `findByNicknameOrderByidDesc`
    - 간단한 조회 적합
    - 복잡한 조건, 동적 쿼리는 JPQL, QueryDSL 사용
    - 코드 가독성 + 생산성 향상
    - 메서드 이름만으로 실행될 쿼리 이해 가능
3. `@Query` : 실제 쿼리문 사용 JPQL
    - 의도 파악 좋음
4. `@EntityGraph`
    - 연관 엔티티 즉시 로딩
5. Pageable : 페이징과 정렬을 쉽게 처리

    ```sql
    Pageable pageable = Pagerequest.of(0, 10, Sort.by("id").descending());
    ```

6. 사용자 정의 레포지토리

#### 주제 4: QueryDSL

동적 쿼리에서 JPQL 사용 시 : 조건에 따라 문자열 추가 오타 시 런타임 에러

- if category → 정렬 분기
- 검색 → 범위, 일치 …
- 문제점
    - 문자열이라 IDE 자동 완성 지원 X
    - @Query 동적 쿼리 불가능

**QueryDSL 사용 시**

- 내부적으로는 JPQL을 써서 성능은 동일함
- 하지만 자바 코드로 쿼리를 작성해서 타입이 안전함 > JPQL은 문자열 기반이라 오타 확인 어렵고 자동완성 지원 제한적이며 동적일 때 불리함
- 페이징이 복잡할 때 사용
- 조인, 집계함수, 동적 필터 조립, 조건 연산
- 코드 재사용성 높음

**Q 클래스**

- 빌드 시 QueryDSL의 메타 모델 클래스가 엔티티들을 Q 클래스로 생성
- JPAQueryFactory와 함께 사용

**BooleanBuilder**

- QueryDSL에서 여러 조건을 조립하거나 동적으로 추가해 where 절을 구성할 수 있도록 도와주는 조건 빌더 객체
- 조건이 많은 동적 쿼리를 if문 없이 간결하게 구성해, 쿼리 가독성과 유지보수성을 높이기 위해서 사용

**where 다중 파라미터**

- QueryDSL에서 where() 절에 여러 조건 메서드를 파라미터로 전달해, null인 조건은 자동으로 제외하는 동적 쿼리 작성 방식
- 조건을 메서드로 분리해 재사용성과 가독성을 높이고, null 조건을 자동으로 제외해 동적 쿼리를 간결하게 작성하기 위해
