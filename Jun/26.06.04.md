    1. 파일을 업로드하기 위해서 고려할 점
        1. 현재 서버 환경 : spring 백엔드 서빙 (로컬), 향후 클라우드 배포 고려 중

            1. 미지원 파일 형식 처리 방식 : 현재 deny → 포맷 변환
        3. 파일 크기 제한 10MB
    2. 압축 및 변환
        1. 파일 포맷 일관성 : jpg와 webp가 동시에 있는 형태로 변환 후 저장
        2. 압축 기능 : 프로필 이미지, 게시글 내부 이미지, 리스트에서 썸네일 몇 % 관리할건지?
    3. 데이터 저장 방식
        1. 스토리지(or 클라우드)에 저장
        2. 파일에 대한 정보는 별도의 image 테이블에 저장
            1. 스토리지 접근 경로
            2. 썸네일 접근 경로
            3. 생성일
            4. 활성 여부(마지막 접근 날짜)
3. 구현하기

    2. 파일 업로드 시 백엔드에서 필요한 작업 : 파일 검증, 포맷 확인 및 처리, 압축 처리, 파일 업로드 처리, 파일 정보 저장

#### 주제 2: 스프링 예외 처리
서비스, 컨트롤러 전부 try-catch 시 중복 코드 발생, 가독성 저하, 유지보수 어려움 → 예외처리 흐름 불명확

- BasicErrorController : 컨트롤러여서 디스패처가 바빠지고, 유의미한 에러 전달 어려움
- `@ResponseStatus` : 에러 HTTP 상태를 변경하는 어노테이션
    - Exception, `@ExceptionHandler`, `@RestcontrollerAdvice` 에서 사용함
    - 응답 자체는 BasicErrorController에 의해 만들어짐 → 응답 내용 수정 어렵고, 예외 클래스 강하게 결합, 외부에서 정의한 Exception 이용 어려움
- ResponseStatusException : 스프링5에서 추가된 예외 응답
    - HTTPStatus와 reason, cause 추가 가능
    - Unchecked Exception 상속 : 명시적 에러처리 안해도 됨
    - new를 통해 빠르게 적용 가능, 예외 클래스 결합도 낮음, 많은 예외 클래스 만들지 않아도 됨
    - 직업 예외처리 프로그래밍 해야하고, 코드 중복이 발생할 수 있음
- `@ExceptionHandler` : 유연하게 어노테이션 추가로 에러처리 가능
    - 컨트롤러가 있는 클래스의 메서드에서 사용
    - Exception 클래스 속성으로 받아 처리 → 어떤 에러처리인지 파라미터로 받음
    - 에러 응답 자유롭게 다룰 수 있음
    - Advice : 스프링 전역으로 적용할 수 있게 함(자바 빈 등록)
        - ControllerAdvice, RestcontrolelrAdvice의 차이점 : `@ResponseBody` 여부
    - 사용법
        - GlobalExceptionhandler 를 만들어 에외처리 담당함

- ResponseStatuse 와 ErrorCode

  **@ResponseStatus 방식**

    ```java
    // 예외 클래스마다 어노테이션 붙여야 함
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public class UserNotFoundException extends RuntimeException {
        public UserNotFoundException() {
            super("유저를 찾을 수 없습니다.");
        }
    }
    
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public class InvalidTokenException extends RuntimeException {
        public InvalidTokenException() {
            super("유효하지 않은 토큰입니다.");
        }
    }
    ```

  **ErrorCode 방식**

    ```java
    // 상태코드 + 메시지를 한 곳에서 관리
    @Getter
    public enum ErrorCode {
        USER_NOT_FOUND(HttpStatus.NOT_FOUND, "유저를 찾을 수 없습니다."),
        INVALID_TOKEN(HttpStatus.UNAUTHORIZED, "유효하지 않은 토큰입니다."),
        BAD_REQUEST(HttpStatus.BAD_REQUEST, "잘못된 요청입니다.");
    
        private final HttpStatus status;
        private final String message;
    }
    
    // 예외는 하나만
    public class CustomException extends RuntimeException {
        private final ErrorCode errorCode;
    }
    
    // 사용
    throw new CustomException(ErrorCode.USER_NOT_FOUND);
    throw new CustomException(ErrorCode.INVALID_TOKEN);
    ```

- ResponseStatusException 이랑 ExceptionHandler

  **`ResponseStatusException`**

    ```java
    // 예외 클래스 만들 필요 없이 각 에러별로 바로 던짐
    throw new ResponseStatusException(HttpStatus.NOT_FOUND, "유저를 찾을 수 없습니다.");
    ```
    
  ---

  **`@ExceptionHandler`**

    ```java
    // 예외 클래스 따로 만들고
    public class CustomException extends RuntimeException {
        private final ErrorCode errorCode;
    }
    
    // 핸들러에서 잡아서 응답 변환
    @ExceptionHandler(CustomException.class)
    public ResponseEntity<ApiResponse<?>> handleCustomException(CustomException e) {
        return ResponseEntity.status(e.getErrorCode().getStatus())
                             .body(ApiResponse.fail(e.getErrorCode().getMessage()));
    }
    ```
    
  ---

  **비교**

  |  | ResponseStatusException | ExceptionHandler |
      | --- | --- | --- |
  | 예외 클래스 | 필요 없음 | 따로 만들어야 함 |
  | 응답 형식 | Spring 기본 형식 | 커스텀 가능 |
  | 메시지 관리 | 던지는 곳마다 직접 작성 | ErrorCode로 중앙 관리 |
  | 적합한 상황 | 간단한 프로젝트 | 실무, 응답 형식 통일 필요 시 |
    - `ResponseStatusException` → 빠르게 던지고 끝, 응답 형식 커스텀 어려움
    - `@ExceptionHandler` → 응답 형식 통일 + ErrorCode로 관리 → **실무 선호** 👍

#### 주제 3: 필터와 인터셉트
**필터**

- 용도
  1. 공통 보안, 인증/인가 작업
  2. 모든 요청에 대한 로깅, 감사
  3. 이미지 데이터 압축 및 문자열 인코딩
  4. 스프링과 분리되어야 하는 기능

- 요청 처리 흐름
  1. 서블릿 컨텍스트에 있는 필터에 도착
  2. 필터 체인을 통해 전처리 수행
  3. 디스패처 서블릿 전달
  4. 컨트롤러를 통해 비즈니스 로직 수행
  5. 응답이 돌아올 때 필터에서 후처리 수행

**인터셉터**
- 용도
    1. 세부적인 인증/인가 작업 (특정 URL, 역할별 접근 제어)
    2. API 호출 로깅, 실행 시간 측정
    3. 컨트롤러로 넘기기 전 데이터 가공
    4. Spring Bean을 활용해야 하는 공통 처리
- 요청 처리 흐름
    1. 디스패처 서블릿이 핸들러(컨트롤러) 조회
    2. 인터셉터의 `preHandle` 실행 (false 반환 시 중단)
    3. 컨트롤러를 통해 비즈니스 로직 수행
    4. 인터셉터의 `postHandle` 실행
    5. 응답 완료 후 `afterCompletion` 실행
