### 새로 배운 내용
#### 주제 1: Inline
데이터 패킷이 반드시 해당 장치를 통과하도록 설계된 네트워크 구조 → 모든 요청이 반드시 거쳐야하는 통로 → 엔진엑스 … 

**특징**

1. 반드시 지나감 (강제성)
    1. 사용자 → 인라인 → 서버
2. 트래픽을 직접 제어함
    1. 정상적인 요청? IP 차단 허용? 이 포트는 허용? 이 요청은 너무 많이 들어오고 있지 않나? 패턴이 악성 패턴인가?
    2. 결과에 따라 → 트래픽 허용, 드롭, 다른곳으로 이동
        1. 차단 : 국가 단위

**사용 이유**

1. 실시간 보안 적용
2. 트래픽 제어 : IP/포트 기반 차단, 프로토콜 제어, Rate Limit, 접근 제어
3. 서버 보호 : 모든 요청을 서버가 처리하지 않음 → 리소스 절약

**대표적인 인라인 장비**

1. 방화벽 : 모든 트래픽이 거치도록 설계 + IPS, WAF, IDS 보안장치
2. 로드밸런서 : 들어오는 요청을 서버들에게 분배

**인라인 장단점**

1. 장점
    1. 강제성, 실시간 차잔, 정책 적용 확실, 로그 확보, 트래픽 제어, 서버 보호
2. 단점
    1. 병목 현상
    2. 단일 장애지점
    3. 잘못된 정책이면 위험함 → 서버 죽음
    4. 장애 극복 방법 : 이중화 → 대기 장비 자동 변환(Failover), 트래픽 우회 통과, 로드밸런싱, 정책 테스트, 모니터링, 규칙 최적화

#### 주제 2: Out-Of-Path
트래픽이 지나가는 길은 두고 실제 트래픽 경로 밖에서 트래픽 복사본을 받아 분석하는 방식

사용 이유

1. 서비스 트래픽에 영향을 주지 않음
    1. 분석 장치가 고장나도 실제 처리 흐름에는 영향을 주지 않음
2. 모니터링 안정적으로 진행 가능
3. 탐지와 분석 집중
    1. 실시간 처리량 증가

특징 : 경로 외부 동작, 비간섭성, 탐지 중심, 서비스 안정성, 확장성 좋음, 직접 차단 한계

**vs Inline**

| 구분 | Inline | Out-of-Path |
| --- | --- | --- |
| 위치 | 실제 트래픽 경로 위 | 실제 트래픽 경로 밖 |
| 원본 트래픽 처리 | 직접 통과시킴 | 복사본을 분석 |
| 차단 능력 | 직접 차단 가능 | 직접 차단은 제한적 |
| 서비스 영향 | 장비 장애 시 서비스 영향 큼 | Sensor 장애 시 서비스 영향 작음 |
| 주요 목적 | 제어, 차단, 필터링 | 탐지, 분석, 모니터링 |
| 대표 장비 | 방화벽, IPS, WAF, 로드밸런서 | IDS, SIEM, Zeek, Wireshark, Flow Logs |
| 비유 | 도로 위 검문소 | 도로 옆 CCTV |

- #### 주제 3 : - WireShark
네트워크 문제, 분석, 소프트웨어 및 통신 프로토콜 개발에 사용

네트워크 캡쳐 도구 사용 이유 : 서버에 요청 로그가 안찍히고, DNS 문제가 아닐때, 네트워크 문제를 보기 위해서

**네트워크 계층 구조**

| Wireshark 표시 | 네트워크 개념 |
  | --- | --- |
| Frame | 캡처된 프레임 전체 |
| Ethernet II | L2, MAC 주소 |
| Internet Protocol | L3, IP 주소 |
| TCP/UDP | L4, 포트 번호 |
| HTTP/DNS/TLS | L7, 애플리케이션 프로토콜 |

**사용법**

필터  : 모든 패킷을 다 보려고 하면 힘듦

- 캡쳐 필터 : 패킷을 캡쳐하기 전에 적용되는 필터. 어떤 패킷을 저장할지 결정
- 디스플레이 필터 : 캡쳐한 다음에 어떤 걸 보여줄지 정하는 필터, 캡쳐한 것들 중 어떤 것을 보여줄지 결정
