#### 주제 1: ElastiCache

인메모리 캐시 서버를 관리해주는 분산형 저장소 서비스

캐시 : 원본 데이터를 매번 다시 조회하지 않도록 자주 사용하는 데이터를 임시로 저장하는 기술

캐시를 왜 메모리에 저장하는가? 자주 접근 되어야 하고 자주 변경됨 → 빠른 속도 필요

캐시 사용 패턴 - Cache-Aside : 앱이 먼저 캐시를 조회하고 없을 때만 DB 조회 후 캐시에 저장
    
TTL : Time To Live

- 캐시 데이터가 자동으로 만료되기까지의 시간
- 캐시에 데이터가 오래 남아있는 걸 방지하기 위해서
    - 메모리를 사용하기에 자원이 한정적
    - 캐시데이터는 정합성을 보장하지 않음

캐시 무효화 전략 : 원본 데이터가 변경됐을 때 기존 캐시 값을 삭제하거나 수정하는 작업

ElasticCache

사용할 수 있는 엔진

| 엔진 | 특징 | 대표 활용 |
| --- | --- | --- |
| Valkey | Redis OSS 호환 계열의 인메모리 저장소 | 캐시, 세션, 랭킹, Pub/Sub |
| Redis OSS | 다양한 자료구조와 기능 제공 | 캐시, 세션, 분산 락, 실시간 기능 |
| Memcached | 단순하고 가벼운 분산 키-값 캐시 | 단순 조회 캐시 |

고가용성 구성

1. Replica : 복제본
2. Multi AZ : 동일하게 나눔
3. Shard : 나누기

캐시를 사용할 때 알아야 하는 문제

1. 캐시는 원본 데이터가 아님
    1. 꺼지면 사라지기 때문에 주의해야함
    2. 복구 정책이 필요함
    3. 캐시가 히트가 안됐을 때 finally → 영구 저장소 조회하는 로직 추가
2. 캐시 스탬피드
    1. 동시에 요청되는 캐시 적재 요청 때문에 지연이 생김
    2. 철회 불가
    3. 임의 값을 추가해서 TTL 시간을 랜덤으로 설정
3. 메모리 부족 현상
    1. 캐시 메모리가 가득 차면 더이상 생성되지 않음 → 새 쓰기 실패

#### 주제 2: ELB

Elastic Load Balancer : 장애 난 서버는 자동으로 제외, 요청을 여러 서번에 나눠 전달하는 관리형 로드 밸런서 서비스

    ```sql
    ELB (Elastic Load Balancer)  ← AWS 로드밸런서 전체 통칭
    ├── ALB (Application LB)     ← HTTP/HTTPS, 7계층
    ├── NLB (Network LB)         ← TCP/UDP, 4계층, 고성능
    └── CLB (Classic LB)         ← 구버전, 거의 안 씀
    ```

**필요 이유** : 해당 서버가 트래픽 때문에 느려지거나 꺼질 수 있는데, 이를 방지하기 위해

- 여러 서버를 사용하더라도 한 서버에만 몰릴 수 있고, 장애난 서버로 계속 요청할 수 있음 → 서비스 품질 저하

**역할** : 요청 분산(처리 성능 높이기), 장애 서버 제외, 요청 분기, 단일 진입점 제공, 서버 확장 지원 … → 엔진엑스, WS도 가능

**헬스체크** : 잘 살아 있는지 확인하는 라우터

- ELB에서 주기적으로 요청

**구조**
1. Load Balancer : 사용자가 접속하는 입구 → Alias LB 사용
2. Listener : ELB가 특정 포트에서 요청을 기다리는 입구
    1. 사용자는 HTTPS로 안전하게 요청, ALB는 서버 내부에서 HTTP로 요청
3. Rule : 어디로 보낼거냐는 규칙, 보통 경로에 따라 타겟 그룹으로 보냄
4. Target Group : 실제 요청을 처리할 서버들의 묶음 (프로토콜 + 포트 기준 전달)

종류 : ALB(HTTP, HTTPS) / NLB(TCP, UDP)

사용자는 캐싱을 바라보게 됨

#### 주제 3: ALB
HTTP, HTTPS 요청을 이해함 → L7(응용) 레이어

URL 경로, 도메인, 헤더 조건 등 … → 얘네 기준으로 요청 분기 가능

구성 방법

1. 보안그룹 구성 : ELB를 구성할 때는 애플리케이션 서버를 인터넷에 노출하지 않는다!
    1. ALB : Inbound는 80, 443(통로) / Outbound는 앱 보안그룹
    2. APP : Inbound는 ALB 보안그룹, 22 / Outbound는 3306 RDS 보안 그룹
2. HTTPS 처리 : HTTPS의 SSL을 제거하거나 붙이는 처리
    1. ALB에서 처리하기 때문에 ALB에 인증서를 붙여야 함 → ACM 통해 발급

장점 : 인증서 중앙 관리, 서버 확장 편리, 자동 갱신 활용, 애플리케이션 단순화

#### 주제 4: CludeFront
전 세계에 분산된 서버 네트워크를 통해 사용자에게 콘텐츠를 빠르고 안전하게 전달하는 서비스

S3만으로 배포하면, 사용자가 많아직나 거리가 멀어지면 로딩 지연, 과한 원본 집중화 등의 문제가 생길 수 있음

사용자 ↔ 원본 서버 사이에 캐시 계층이 CDN(가까운 곳에서 받아옴)으로 분산되어 있어 데이터를 빠르게 가져올 수 있음 → 엣지에는 복제본이 있음
**구성 요소**

| 구성 요소 | 역할 |
  | --- | --- |
| **Distribution** | CloudFront 배포 단위. 어떤 원본과 캐시 정책을 사용할지 정의 |
| **Viewer** | CloudFront에 요청하는 사용자 또는 브라우저 |
| **Edge Location** | 사용자 가까이에서 캐시된 콘텐츠를 전달하는 위치 |
| **Origin** | 원본 콘텐츠가 실제 저장된 위치 |
| **Cache Behavior** | 어떤 경로를 어떤 원본으로 보내고, 어떻게 캐시할지 정하는 규칙 |
| **Cache Policy** | 캐시 키와 TTL 등 캐시 동작을 결정하는 정책 |
| **OAC** | S3 원본에 CloudFront만 접근하도록 제한하는 기능 |

Cache Behavior : URL 경로에 따라서 어떤 원본으로 요청을 보내고 캐시를 어떻게 적용할 지 정하는 규칙. 특성에 따라 전략 선택 필수

어떤 컨텐츠를 캐시하나?

| 콘텐츠 종류 | 예시 | 권장 캐시 전략 |
  | --- | --- | --- |
| 변경이 거의 없는 정적 파일 | 로고, 폰트, 라이브러리 파일 | 길게 캐시 |
| 빌드 결과물 | `app.8f3a2.js`, `main.2c4d.css` | 매우 길게 캐시 |
| 일반 이미지 | 프로필 이미지, 게시글 이미지 | 변경 빈도에 맞춰 캐시 |
| HTML | `index.html` | 짧게 캐시 |
| 로그인·결제·개인 정보 API | `/api/me`, `/api/orders` | 기본적으로 캐시하지 않음 |
| 공개 조회 API | 상품 목록, 공지 목록 | 조건부로 짧게 캐시 가능 |

CloudFront vs ELB

- 사용자 요청 앞에 위치
- 해결하려는 문제가 다름 → 프론트/백, 캐시사용여부 …

CloudeFront + ALB
실무

1. 정적 웹 사이트 : S3에 Vite, React 프론트 엔드 빌드 일 저장 시
2. 이미지 업로드 서비스

장단점

- 이미 캐싱되어 있는 이미지 사용 가능
- 파일에 대한 원본 콘텐츠를 사용자 가까이에서 빠른 전달 → 원본 서버에 반복 요청을 줄일 수 있음
