### 새로 배운 내용
#### 주제 1: 서버리스
운영의 상당 부분을 AWS가 책임지는 방식

람다 방식 : 함수 코드 작성, 이벤트 트리거 연결, IAM 권한 설정 → AWS가 알아서 관리

**EC2 vs Lambda**

- EC2는 서버를 계속 켜두는 방식이며, 보통 시간단위 과금
    - 바로 응답 가능, 서버 내부 상태 관리하기 쉬움, 필요한 라이브러리 자유롭게 사용 가능
    - 요청이 없어도 비용 발생
- Lambda는 이벤트가 발생하면 실행됨
    - 실행 시간 만큼 과금, 확장성 좋음
    - 일정 시간이 지나면 사라짐

이벤트 : 누군가 버튼을 누르면 실행되는 코드

**예시**
    
| 이벤트 소스 | Lambda 사용 예시 |
| --- | --- |
| API Gateway | HTTP API 요청 처리 |
| S3 | 이미지 업로드 후 썸네일 생성 |
| EventBridge | 매일 오전 9시 정기 작업 |
| SQS | 비동기 메시지 처리 |
| DynamoDB Streams | 데이터 변경 감지 후 후속 처리 |
| CloudWatch Alarm | 장애 발생 시 자동 복구 작업 |
| Cognito | 회원가입 전후 사용자 처리 |
| ALB | HTTP 요청을 Lambda로 전달 |

**순서**

1. 이벤트 발생
2. 람다가 실행 환경 준비 : 격리된 실행공간에서
3. init : 초기화 코드들 실행 (함수 + Lambda layer 다운로드)
    1. 파이썬, 스크립트로 주로 실행됨 → Runtime 초기화
    2. 핸들러 밖에 작성한 코드들도 초기화
    - 최초 실행 (Cold Start) : 실행 환경이 준비된 게 없어서 초기 세팅하느라 시간이 걸리는 것 → 처리시간 증가 → 람다 안에서 warmup 설정(특정 시간마다 health check)
    - ↔ Warm Start : 기존에 만들어진 실행 환경 재사용 → 핸들러만 붙이면 돼서(init 건너뜀) 요청 처리시간이 짧음
4. Invoke 단계 : 핸들러 호출

언제 쓰는가?

1. 파일 업로드 처리 : S3 업로드 완료 트리거 발생 시 ex) 리사이징, 형태 변환, 바이러스 검사 등 …
2. 극초기 API : 하지만 API는 응답 시간이 중요하기 때문에 비추
3. 예약 작업(배치 작업) : 매일 새벽, 임시 데이터 삭제
4. 비동기 작업 처리 : 알람 등을 비동기로 처리할 때

람다는 실행 환경 재사용될 수 있으나 저장된 값은 언젠가 무조건 사라짐 → 전역변수 X, /tmp 저장 X → 외부 저장소 사용 필수

**람다 설정**

1. 메모리 : 메모리가 늘어나면 자동으로 CPU 할당도 늘어남
2. 타임아웃 : 얼마동안 실행될 수 있는지 설정. 람다 함수는 15분 이상 실행되지 않음

**실무**

1. 람다 단일 책임 원칙 (SPR)
    1. 각 단계의 람다를 만들고, 다음 단계 이벤트 핸들러 호출
    2. 독립적인 처리가 필요한 경우 함수 쪼개기
2. cold start 문제 대응하기
    1. 실행 환경이 무거운 코드 쓰지 않기
        1. 자바 코드는 런타임이 무거움
    2. 프로비저닝 : 람다 실행환경을 미리 준비해두는 것
        1. 람다 함수는 항상 N개 이상의 실행환경을 준비해두는 걸 설정
    3. 헬스체크 : 이벤트 브릿지한테 N초마다 호출
3. 동시성 문제 해결
    1. limit까지 자동 확장
    2. AWS 한 계정에 한도가 있는데, 람다는 자동으로 확장되기 때문에 무한대로 할당받을 수 있음 → DB 커넥션을 차지해서 DB가 뻗어버리는 문제 → 큐로 해결
4. 장기간 연결 요청 피하기
    1. 최대 15분까지 실행됨
5. 람다를 모든 백엔드 기능의 서버로 두지 않기
    1. 모든 API는 항상 람다랑 어울릴 수 없음을 알고있기
    2. DB 연결을 지속적으로 해야할 수도 있음

#### 주제 2: API Gateway

클라이언트의 API 요청을 받아 백엔드로 전달하고, 인증, 권한 요청제한, 모니터링 등의 공통 기능을 API 앞단에서 관리하는 AWS 완전관리형 서비스

백엔드 서비스가 커지면, API 기능 + 공통 문제 해결 필요

공통문제는 여기서 common에 포함되는 로직이나, CORS, 재요청 …

→ 비즈니스 로직 처리도 바쁜데 API 입구 역할도 함 → 이 역할을 API Gateway 로 위임

사용 이유

1. API 입구를 하나로 줄이기 위해서
    1. MSA(서비스 분리)가 되면 필요함
2. 인증 인가 분리할 때
    1. 서버를 적절한 스택들로 분리한 코드들에는 전부 인증, 인가가 있음
    2. API GW에 인증인가 책임을 전가함
3. 요청 수 제한 API Key 컨트롤
    1. 외부에 API 제공 시 횟수 컨트롤
4. CORS 중앙 집중 관리
5. 람다같은 AWS 서비스를 직접 호출할 수 있음

유형

1. HTTP API : 단순하고 비용 효율적인 코드, ALB로도 처리 가능
2. REST API : 보통 처리. 기능이 많고 세밀한 제어 가능
3. WebSocket API : 양방향 실시간 연결

#### 주제 3: CloudWatch
CloudTrail : AWS 계정에서 발생하는 모든 API 호출과 콘솔 활동을 자동으로 기록하고 추적하는 서비스

모니터링은 운영의 기본 장치이다. 그를 도와주는 게 CloudWatch

**핵심 기능**

1. Metrics : 시간에 따른 숫자 지표 확인 (CPU, 요청 수, 응답별 수)
    1. 자주 보는 메트릭
        1. EC2(CPU 사용률, 인스턴스 상태)
        2. ALB(500 에러, 응답 시간)
        3. RDS(DB 연결 수, 남은 스토리지)
        4. ElastiCache(메모리 부족, 캐시 효율)
    2. 문제가 발생했을 때 알려줌
2. 로그
    1. 문제가 왜 생겼는지 기록
    2. 중앙 집중화 가능
3. 알람 : CPU 80% 초과 시같이 특정 조건 시 알림 또는 대응
4. 대시보드 : 여러 지표를 한 화면에서 확인
5. 로그 인사이트 : 로그를 쿼리로 분석

#### 주제 4: 옵저빌리티
시스템 내부를 열어보지 않더라도 시스템이 밖으로 남기는 로그들을 통해 무슨일이 어떻게, 왜 일어났는지 추론하는 능력

필요 이유 : 서버가 적으면 ssh 접속으로 해도 되지만, 많으면 어려움

모니터링은 이미 예상한 문제를 방지하고 확인하는 것이고, 옵저빌리티는 예상하지 못한 문제도 데이터로 원인 추론 가능

메트릭, 로그, 트레이스(지나간 흐름 및 걸린 시간) 잘 보기

- Trace ID : 하나의 사용자 요청 추적을 위해 ID를 붙임

높이는 방법

1. 로그 구조화하기
2. 핵심 API 응답시간/오류율 수집 : 중요기능(회원가입, 결제, 작성 등)
