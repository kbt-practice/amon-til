### 새로 배운 내용
#### 주제 1: 운영체제
하드웨어, 소프트웨어를 관리. 사용자와 컴퓨터 간 상호작용을 도움

자원 관리, 사용자 편의성, 작업 조율(멀티태스킹), 보안 및 안정성 때문에 필요함

주요 계층

1. 사용자 인터페이스 : GUI, CLI
2. 시스템 콜 : 응용 프로그램이 운영체제를 통해 서비스에 접근할 수 있도록 하는 인터페이스
    1. 유저모드에서 커널에게 기능을 요청하는 통로
    2. 유저모드 → 시스템 콜 → 커널 모드 → 결과 반환 → 유저 모드
3. 커널 : 하드웨어 ↔ 프로그램 사이에서 중재자
    1. 유저 모드 : 우리가 직접 사용하는 프로그램이 실행 되는 공간 ex) 자바 프로그램, 줌, 브라우저
    2. 커널 모드 : 운영체제의 핵심 기능들이 동작하는 공간 ex) 파일 열기, 메모리 쓰기, 네트워크 전송, DISK IO
    - 유저 모드는 하드웨어 접근 제한 → 커널에게 부탁해야함 → 시스템 콜에서

리눅스 : 일부 무료로 사용할 수 있는 안정성 높고 가벼운 오픈소스 운영체제. CLI 제어 가능

#### 주제 2: 메모리
- 구조
    
    메모리를 4가지 구역으로 나누어 관리
    
    | **영역** | **역할** | **저장되는 데이터** | **특징** |
    | --- | --- | --- | --- |
    | **코드(Code) 영역** | 프로그램의 실제 명령어가 저장되는 공간 | 우리가 작성한 소스 코드, 제어문, 함수 | **읽기 전용(**실행 중 코드가 변경되는 것 방지) |
    | **데이터(Data) 영역** | 프로그램이 시작될 때부터 끝날 때까지 유지되는 데이터의 공간 | 전역 변수, 정적 변수(Static) | 프로그램 실행과 동시에 할당되고, 종료 시 소멸 |
    | **힙(Heap) 영역** | 필요할 때마다 동적으로 빌려 쓰고 반납하는 자유 공간 | 파일 내용, 사용자가 입력한 긴 텍스트, 자바의 객체 | 런타임에 크기가 결정 |
    | **스택(Stack) 영역** | 잠깐 사용하고 버릴 임시 데이터가 쌓이는 작업 공간 | 지역 변수, 매개 변수, 함수 호출 내역 | 함수가 끝나면 **자동 삭제** |
    - 안정성 확보, 자원 관리, 유연성을 위해
- 관리자
    
    디스크에서 필요한 데이터를 메모리에 올려서 사용함 (디스크는 느려서)
    
    CPU ↔ Memory ↔ DISK ← 메모리에서 작업하는 주체가 메모리 관리자
    
    제한된 자원을 최적으로 사용, 프로세스 간 충돌 방지, 물리적인 메모리의 한계 극복(+가격)
    
- 주소 체계
    
    데이터를 메모리에 올리는데, 그 위치를 알기 위해 부여한 고유한 주소
    
    메모리 주소가 동작하는 방식 : 운영체제가 관리하는데, 브라우저에서 접근 못하게 함
    
    1. 논리 주소 : 프로그램이 바라보는 메모리 주소
        1. 실제 아님. 프로그램은 메모리를 자기가 0번부터 사용한다고 생각함
    2. 물리 주소 : 실제 메모리 주소
    3. MMU : 메모리 관리 장치
        1. CPU가 어디 위치에 있는 걸 가져오라고 명령하면 MMU가 가상 주소에 해당하는 물리주소 변환해줌
- 관리
    
    **오버레이** : 코드를 모듈로 쪼개고 필요한 조각만 올려서 사용하는 방식
    
    - 개발자가 코드를 짤 때 메모리에 올라갈 모듈을 분리하고, 메모리 공간을 고려함 → 개발 난이도 상승
    
    **스왑** : 사용하지 않는 데이터를 디스크의 스왑 공간으로 이동해 RAM 메모리를 확보하고, 필요 시 다시 메모리로 복구
    
    - 속도가 상당히 느려짐
    - 스레싱 : 메모리가 많이 부족하면 스왑 아웃/인 하는 객체가 많아짐 → 스왑 자체가 프로세스 → 컴퓨터가 멈춘것처럼 보임
    - 위 문제 때문에 쿠버네티스, 도커 환경에서는 스왑 사용이 안됨
    
    | **구분** | **오버레이** | **스왑** |
    | --- | --- | --- |
    | **관리 주체** | **프로그래머** | **운영체제** |
    | **관리 단위** | 하나의 프로그램 내부의 **모듈** 단위 | 실행 중인 **프로세스** 단위 |
    | **목적** | 단일 프로그램의 크기 제한 극복 | 여러 프로그램을 동시에 실행하기 위한 공간 확보 |
- 분할
    
    여러 프로그램이 한정된 메모리 자원을 같이 사용하는 다중 프로세스 환경일 때, 메모리 자원을 충돌없이 쓰기 위해 메모리 공간을 쪼개서 사용함
    
    1. 고정 분할 방식
        1. 메모리를 사전에 고정된 크기의 블록으로 나눔
        2. 블록 1 : 1 프로세스
        3. 관리 용이, 프로세스 할당 속도 빠름, 프로세스 간 충돌 가능성 낮음
        4. 내부 단편화 : 할당된 메모리 블록보다 프로세스가 작아서 버려지는 빈 메모리 공간
    2. 가변 분할 방식
        1. 프로세스의 실제 크기에 딱 맞춰 메모리를 동적으로 할당 → 1번의 문제점 해결
        2. 외부 단편화 : 프로세스들이 들어갔다 나왔다 하면 중간에 작은 공간들이 남음. 전체 공간은 충분하나 연속된 공간이 부족하게 됨
            1. 비어있는 공간 재정렬 → 컴퓨터가 멈춤(by JVM GC)
    3. 페이징 방식
        1. 메모리와 프로그램을 같은 크기로 잘라 저장 → 2번 문제점 해결
        2. 연속적으로 비워둘 필요가 없음
        3. 페이지 테이블을 사용하기 위해 메모리를 추가로 사용, 주소 변환 시 시간(MMU)
    4. 세그멘테이션 방식
        1. 프로그램의 논리적인 단위로 잘라 연속된 공간에 보관
        2. 프로그램 구조 그대로 반영, 데이터 보호 및 공유 편리
        3. 세그먼트 크기 제각각이어서 외부 단편화 발생 가능
        4. 컴파일러 by 링커
    - 현대는 페이징 + 세그멘테이션
        - 세그먼트 방식 분리 → 페이징으로 나누기
- 가상
    - 목적 : 물리적인 RAM 보다 큰 프로그램을 실행하면서 + 동시에 여러개의 프로세스를 안전하게 실행
    - 페이징 + 세그멘테이션
        - 가상 메모리 - 배달 서비스 거대한 비즈니스 개념
        - 페이징 - 그 서비스를 실현하기 위한 방법 (오토바이 배달)
#### 주제 3: 프로세스와 스레드
프로세스 : 실행 중인 프로그램. 운영체제에게 자원을 할당받아 독립적으로 실행되는 것의 단위 → 컴퓨터가 여러개의 프로그램을 어떻게 돌리는지를 이해

스레드 : 프로세스 내 병렬 처리를 위해 생성되는 실행 흐름의 최소 단위 → 동시성 + 병렬성을 활용해서 프로그램의 실행 속도와 반응성 극대화

- 멀티 스레드 : 여러개의 스레드. 대표적으로 Web Spring이 있음
    - 요청마다 스레드 생성
    - 대규모 개발 시 EJB
- 싱글 스레드 : 하나의 스레드. 이벤트 기반. 대표적으로 Node.js가 있음
    - 적은 리소스로 많은 응답
    - 객체지향적이지 않아 코드의 재사용성이 떨어지고 대규모 개발이 힘들어짐
- 노드는 싱글 스레드이지만 사실은 JS가 싱글 스레드이고 워커 스레드(보통 4개)가 고정적으로 존재한다. 자바는 동적으로 스레드를 만들 수 있다는 있다는 게 장점임.
#### 주제 4: IPC
프로세스 간 운영체제의 자원을 요청/접근할 때 제공하는 것

1. 메세지 전달 : 커널이 가운데에 껴서 메세지를 전달해줌
2. 공유 메모리 : 프로세스 간 공유 가능한 공간, 접근하는 프로세스 많은 경우 동기화 문제 발생할 수 있음
- 메시지 전달: 파이프, 소켓, 큐 등
- 공유 메모리: 뮤텍스, 세마포어로 동기화

#### 주제 5: PCB(프로세스 제어 블록)
운영체제가 각 프로세스를 효율적으로 관리하기 위해 부여한 정보들이 담겨있음

CPU는 모든 프로세스를 기억하지 않음 → 어디까지 했는지 모름 → 이를 PCB에 저장해서 프로세스 전환(컨텍스트 스위칭) 시 확인
