- 데이터를 주고받는 게 핵심

- API, DB 응답 확인 (CORS)
- 서버를 외부에 공개 : 공인 IP/Port, 라우팅 …
- 사내망과 외부망 분리
- 클라우드 구성
- 보안 설계

#### 주제 3: Node
네트워크에서 데이터를 생성/수신/전송하는 장치나 지점

사용 이유 : 노드가 있어야 네트워크에서 데이터가 흐름

종류 : 클라이언트, 서버, 라우터, 스위치 …

특징 : 주소를 가짐

- Mac : 네트워크 인터페이스 카드에 부여된 물리적인 식별자 - 기기의 고유 일련번호
- IP : 네트워크 상에서 위치를 나타내는 논리적인 주소 - 도로명 주소
    - Mac만 존재하면 같은 네트워크 안에서만 사용한다는 한계 존재
        
        → 다른 네트워크랑 통신 시 IP 주소 사용
        
        → 같은 네트워크 안에서 host를 찾을 때 Mac 주소 사용
        

Host : 사용자를 대신해 실제 데이터를 생성하거나 소비하는 장치

- node랑 다른 점? 노드는 지점이고 호스트는 사용하는 최종 장치(출발지, 목적지)

실무에서 노드 장애 : 로드밸런서, API 서버, 프록시 서버, 클라이언트 브라우저, 데이터베이스

클라우드에서 노드 장애 : AWS(클라우드프론트, WAF, ALB, EC2, RDS, 엘라스틱캐시)

#### 주제 4: LAN

제한된 공간 안에서 여러 컴퓨터와 장치를 연결한 내부 네트워크

인터넷이 없더라도 내부망으로 통신(노드들 간 통신)하면 LAN이라고 함

사용 이유 : 빠름, 자원 공유, 관리 편의성, 비용 절감

구성

| 구성 요소 | 역할 |
| --- | --- |
| **PC, 서버, 프린터 등** | 네트워크에 참여하는 장치 |
| **NIC** | 장치가 네트워크에 연결되기 위한 네트워크 인터페이스 |
| **스위치** | 같은 LAN 안의 장치들을 연결하고 데이터를 적절한 장치로 전달 |
| **라우터/공유기** | LAN과 외부 인터넷을 연결 |
| **이더넷 케이블 또는 Wi-Fi** | 장치들이 데이터를 주고받는 물리적/무선 연결 방식 |
| **IP 주소** | LAN 안에서 장치를 구분하기 위한 논리적 주소 |
| **MAC 주소** | 같은 네트워크 구간에서 장치를 식별하기 위한 물리적 주소 |

알아야하는 것

- 같은 네트워크(IP) 대역대를 가짐 : CIDR, Address - IP 앞부분이 똑같음
- LAN 내부 통신 ≠ 인터넷 통신 : 같은 네트워크 안에서 통신 vs 다른 네트워크와 통신
    - 랜 통신에는 라우터 필요없음

실무

- 같은 와이파이 사용 시 → localhost 로 띄운 서버 접속 가능 → 테스트 서버 필요 없음 → 보안 굿~ 통신 속도 굿~ 비용 저렴!
- 데이터베이스 접속 실패 시 → 서버(EC2)내에서 디버깅 할 때
- LAN 설계 시 네트워크 설계의 출발지점

WLAN(무선랜) : 물리 케이블이 아닌 무선 신호 기반(WIFI)으로 통신하는 LAN (어떻게 연결?)

- 사용 이유 : 이동성, 설치 편의성, 공간 활용, 다양한 장치 지원
- 구성 : 무선 단말, 무선 NIC, AP(Access Point), SSID, 인증/암호화
- 주의사항 : 신호 간섭, 보안 문제, 속도 변동, AP 과부하, 음영 지역
    - 신호 간섭이나 음영 지역에서 처리 → Retry 횟수, Timeout …

VLAN : 하나의 물리적인 네트워크를 여러개의 독립된 네트워크처럼 나누는 기술 (어떻게 나눔?)

- 사용 이유 : 보안 강화, 트래픽 감소, 관리 편의성, 비용 절감
- 구성 : 스위치, 라우터
- 주의사항 : 정책 필요, 통신 별도 설정, 설정 실수 주의, 운영 복잡도 증가, 게스트망 분리 필수
- 브로드캐스트 : 같은 네트워크 안에서 모든 장치에게 동시에 메세지를 전송하는 것
    - 내 컴퓨터는 본인의 IP 주소가 뭔지, 프린터가 어디있는지 모름 → 통신 불가
        - 초기에 연결된 컴퓨터는 브로드캐스트를 통해 새로 등록되었음을 알림

| 구분 | LAN | WLAN | VLAN |
| --- | --- | --- | --- |
| 전체 이름 | Local Area Network | Wireless Local Area Network | Virtual Local Area Network |
| 한 줄 정리 | 가까운 공간의 내부 네트워크 | 무선으로 연결하는 LAN | 물리 네트워크를 논리적으로 나누는 기술 |
| 핵심 질문 | 어디까지가 같은 내부망인가? | 랜선 없이 연결할 수 있는가? | 같은 장비를 쓰면서 네트워크를 나눌 수 있는가? |
| 연결 방식 | 주로 유선 이더넷 | Wi-Fi 같은 무선 신호 | 물리 연결과 별개로 논리 분리 |
| 주요 장비 | 스위치, 라우터, 케이블 | AP, 무선 공유기, 무선 NIC | VLAN 지원 스위치, L3 스위치, 라우터 |
| 목적 | 내부 장치 간 통신과 자원 공유 | 이동성과 무선 접속 제공 | 보안, 트래픽 분리, 관리 편의성 |
| 예시 | 사무실 PC와 프린터 연결 | 강의실 Wi-Fi, 카페 Wi-Fi | 학생망/교직원망/게스트망 분리 |

**PAN < LAN < MAN < WAN**

WAN : 지리적으로 멀리 떨어진 네트워크를 연결하는 광역 네트워크

- 인터넷이 여기에 해당
- 사용 이유 : 원거리 통신, 글로벌 서비스 제공, 본사-지사 연결, 클라우드 연결

MAN : LAN보다 넓고, WAN보다는 좁은 범위를 연결하는 네트워크

- 여러 건물 간 네트워크 → LAN의 집합
- 사용 이유 : 여러 건물 연결, 지역 단위 통합

(LAN)

PAN : 개인 주변의 가까운 장치들을 연결하는 네트워크

- 사용 이유 : 개인 장치 연결, 짧은 거리 통신, 편리한 사용성, 저전력 통신

#### 주제 5: 네트워크 장비
배우는 이유 : 여러 네트워크 장비가 협력해서 통신하기 때문에. 어디서 에러가 나는지 파악하기 위해서.

기본적인 흐름

```
[내 컴퓨터]
  └─ NIC: 네트워크에 연결되는 입구
      └─ MAC 주소: 같은 네트워크 안에서 나를 구분하는 주소
          └─ 스위치: 같은 LAN 안에서 목적지 장치로 전달
              └─ 라우터: 다른 네트워크로 가는 경로 결정
                  └─ 게이트웨이: 내부망과 외부망을 연결하는 관문
                      └─ 인터넷
```

- NIC
    
    네트워크 장치가 네트워크에 연결되도록 해주는 하드웨어 구성 요소
    
    호스트랑 네트워크 사이 데이터를 주고받는 역할
    
    1. 유선 : 고정 장치(컴퓨터, 서버)
        1. 연결 시 이더넷 케이블, 랜선 → 안정적
    2. 무선 : 와이파이 연결할 때
        1. 이동성 있는 장비(노트북, 스마트폰)에 무선 NIC
- MAC Address
    
    MAC Address : NIC에 부여된 고유한 물리적 식별자
    
    - 00:1A:2B:3C:4D:5E
        - 앞에 3개는 제조사 뒤에 3개는 장치 식별 부분
    - NIC 제조될 때 같이 하드웨어에 내장됨
    - 와이파이가 변경되면 IP가 변경되는데 MAC Address는 유지
    - ARP (Address Resolution Protocol) : IP 주소에 해당하는 MAC 주소를 알아내는 프로토콜
    - 맥 주소는 같은 네트워크 구간에서 전달할 때만 사용한다.
    
    사용 이유 : LAN 안에서 어떤 장치에게 데이터 전달 시 판별하려고 (같은 네트워크 구간에서만 사용, 외부 네트워크 통신 시 직접 사용X)
    
    맥은 보통 NIC 안에 들어있음 → 호스트 1 : 1 MAC 비율로 존재
    
    **vs IP**
    
    | 구분 | MAC 주소 | IP 주소 |
    | --- | --- | --- |
    | 의미 | 네트워크 인터페이스의 물리적 식별자 | 네트워크 상의 논리적 위치 주소 |
    | 사용 계층 | 데이터 링크 계층, L2 | 네트워크 계층, L3 |
    | 주 사용 범위 | 같은 LAN 내부 전달 | 다른 네트워크까지의 경로 결정 |
    | 변경 가능성 | 기본적으로 하드웨어에 부여됨, 단 스푸핑 가능 | 네트워크 환경에 따라 변경 가능 |
    | 예시 | `00:1A:2B:3C:4D:5E` | `192.168.0.10` |
    | 주로 보는 장비 | 스위치, 브릿지, AP | 라우터, L3 스위치 |
    
    **스푸핑** : 노드의 원래 MAC 주소를 다른 MAC 주소처럼 보이게하는 기술 (공유기 가로채기)
    - 테스트, 장치 교체, 접근 우회, 추적 회피
    
    **맥 주소로 이슈 처리하기**
    1. 같은 사무실 서버에 접속이 안됨
    2. 도커, VM 네트워크 문제 : 가상 NIC, MAC 생성
    3. 로컬 테스트 환경
    4. 로드밸런서/프록시 뒤 통신
    5. 보안 로그 분석 : MAC 주소로 장치 추적 가능
- 리피터
    
    약해진 신호를 증폭/재생 시켜주는 도구 ex) 신호 중계기
    
    사용 이유 : 신호 감쇠 방지, 네트워크 거리 확장, 통신 안정성 향상, 물리적 제약 보완
    
    한계 : 트래픽 구분 불가, 불필요한 신호도 전달, 네트워크 충돌 문제 해결 불가, 보안 제어 불가
    
- 게이트웨이
    
    서로 다른 네트워크를 연결하여 통신을 가능하게 하는 장치
    - 내-외부 통신을 연결
    종류
    
    1. 기본 게이트웨이 : 모든 외부와 연결됨
    2. API 게이트웨이 : 특정 API만 실행하게 문이 있음
- 라우터
    
    서로 다른 네트워크를 연결하고, IP 주소를 보고 패킷이 갈 다음 경로를 결정해주는 장치
    
    1. 다른 네트워크로 보내는 장비
        1. ↔ 스위치 : 같은 LAN 네트워크 안에서 장치 연결
        2. 네트워크가 외부로 이동할 때 스위치로는 부족함 → 라우터 사용
    2. IP를 보고 판단 : IP 주소를 보고 패킷을 어디로 라우팅할지 결정 (L3)
        1. 스위치는 MAC 주소를 보고 전달 (L2)
    
    인터넷은 다양한 네트워크의 집합 → 서로 연결되어있음 → 서로 다른 네트워크가 연결되어야 함 → 라우터를 통해 연결
    
    동작 방식
    
    1. 패킷을 받음 : 출발지/목적지 IP
    2. 패킷을 열어 목적지 IP 확인
    3. 내부라우터는 내부 라우터가 아닌 건 전부 외부인 것 + 외부로 보내려면 0.0.0.0/0 으로 보내야함을 알고있음
    4. 실제로 경로가 되는 외부 라우터 → 라우팅 테이블을 확인한다.
    
    라우터와 게이트웨이 : 같은 장비 안에 들어있음. 게이트웨이는 출구이고, 라우터는 어디 게이트웨이로 갈 지 결정
    
    - 패킷 → 네트워크 계층에서 다루는 데이터의 단위
    - 라우팅 → 목적지까지 패킷이 가도록 경로를 결정하는 과정
    - 라우팅 테이블 → 라우터가 참고하는 길 안내표
    - 기본 경로 → 라우터가 경로를 못찾으면 보내는 경로
    
    | 방식 | 설명 | 예시 |
    | --- | --- | --- |
    | **직접 연결 경로** | 라우터 인터페이스에 IP를 설정하면 자동 등록 | LAN, WAN 인터페이스 |
    | **정적 라우팅** | 관리자가 직접 경로 입력 | 특정 사내망은 VPN으로 보내라 |
    | **동적 라우팅** | 라우터끼리 경로 정보를 교환 | OSPF, BGP |
    
    OSPF : 조직 내부에서 라우터끼리 경로 공유
    
    BGP : 조직과 조직 사이에서 경로 공유
    
- 허브
    
    데이터를 여러 장치로 전달해주는 장치
    
    네트워크를 만들려면 : 연결 → 통신
    
    - 중간에 전달해주는 매개체가 있어야 편함
    - 스위치랑 라우터는 비용이 비쌈
        - 라우터는 테이블 알아야하고, 홉 알아야하고, 비용 계산 …
        - 스위치는
        - 단순한 통신을 위해서 사용
    - 여러 장치를 하나의 네트워크에 연결하고, 데이터를 열어보지 않은 채 모든 장치에게 뿌려줌
    - 싸고, 구성이 단순함
    - 불필요한 트래픽이 많아지고, 보안이 별로, 충돌 발생
        - Collsion Domain : 둘 이상의 장치가 동시에 데이터를 보내면 신호가 섞이는 현상
- 스위치
    
    연결된 모든 장치에 전달하는 허브의 문제점
    
    → 같은 네트워크 안에서 MAC 주소를 보고 해당 장치에게만 보냄
    
    사용 이유 : 정확한 데이터 전달, 불필요한 트래픽 감소, 충돌 감소, 성능 향상, 보안성 향상, VLAN 확장 가능(스위치에서 작동함)
    
    동작 방식
    
    1. 스위치가 목적지 Mac 주소를 알 때 : MAC Address Table 확인 후 전달
    2. 모를 때(Flooding) : 일단 모르는 MAC 주소의 모든 포트로 보냄(Unknown Unicast Flooding) → 받은 애들은 본인 게 아니라 응답 → 한명이 작업 → 스위치에 응답 → 맥 주소 테이블 업데이트
        1. ≠ 브로드 캐스트 : 목적지가 전체인지 하나인지 구분
        2. ≠ 허브 : 신호 기반인지 MAC 주소 기반인지
        3. ≠ 라우터 : 네트워크가 동일한지 다른지
- 브릿지
    
    두 개 이상의 LAN 세그먼트를 연결하는 장치 BUT 모든 데이터를 연결하지 않음
    
    사용 이유 : LAN 연결, 불필요한 트래픽 감소, 
    
    LAN이 커지면 VLAN으로 쪼개는데, VLAN은 스위치에서 동작하기 때문에 스위치가 부하를 받음 → LAN을 여러개 두자 → 이를 연결하는 게 브릿지
    
    ≠ 스위치 : 브릿지는 소프트웨어 기반, 스위치는 하드웨어 기반
