#### 주제 3: ORM

자바의 객체와 데이터베이스 테이블을 자동으로 연결, 변환, 조작할 수 있게 해주는 기술

왜 사용?

- 객체는 객체지향으로
- DB는 테이블 중심으로 설계

→ 두 개를 연결

객체지향 vs 관계형 DB

1. 상속과 다형성
    1. 자바(객지) : 클래스로 상속 지원 O
    2. RDB : 상속의 개념이 없음. 유사하게 만들 수는 있으나 복잡함
2. 연관관계
    1. 자바 : 참조 관계 → 단방향
    2. RDB : 외래키 + JOIN → 각 테이블간의 관계 설정 가능
3. 데이터 탐색 범위
    1. 자바 : 그래프 탐지. A가 있을 때, A의 부모는 B, B의 부모는 C → 한 단계씩
    2. RDB : SQL에 따라 넓게 탐지 가능
4. ORM
    1. 자동 매핑, 객체 모델링

ORM 사용법

- RDB로 테이블 제작 → 하이버네이트가 자동으로 SQL 생성 → JPA(ORM) 구현체 엔진
    - 하이버네이트 : JPA 설계도를 통해 움직임. 자동으로 SQL 생성
    - JPA : Java Persistance API
- 자바 객체 (Entity) : 매핑 설정 - `@Entity @Column @Id @NotNull`

#### 주제 4: JPA

자바 진영에서 공식적으로 채택한 ORM 표준

JDBC : Java Databse Connectivity

- 자바 - 데이터베이스의 표준 어댑터
- 데이터베이스에 쿼리를 전달할 수 있는 통로
- 개발자가 매번 연결, 해제, raw SQL 직접 작성

JPA : Java Persistence API

- 직접 JDBC를 다루지 않음 → ORM(대리인)
- 동작 : 개발자가 엔티티 전달 → 구조를 분석해서 SQL 생성 → JDBC를 통해 실행

#### 주제 6: 영속성 컨텍스트

JPA가 엔티티 객체를 관리하며 동일 트랜잭션 내에 변경 사항을 추적하는 것

저장소마다 속도차이가 남 > 생각보다 큼 > 고려해야 성능을 낼 수 있음

특징

1. 1차 캐시 : 엔티티-식별자 맵 형태로 저장되어있는 캐시
    1. 엔티티가 있으면 DB 접근을 안 함
2. 쓰기 지연 : 엔티티 인스턴스에 대한 쿼리를 바로 DB로 보내지 않음
    1. 1차 캐시에만 반영하고, 쿼리는 쓰기 지연 저장소에 쌓아둠
    2. 변경사항은 동기화할 때 한번에 적용 > 동기화가 안돼서 무결성이 깨지는 경우 발생
3. 변경 감지
    1. 1차 캐시의 상태와 비교하여 스냅샷 추가
4. 트랜잭션
    1. 영속성 컨텍스트와 트랜잭션은 생명주기가 똑같음
    2. 흐름
        1. 트랜잭션 시작
        2. 영속성 컨텍스트 생성
        3. 엔티티 변화 감지
        4. 작업 완료, 트랜잭션 커밋
        5. 용속성 컨텍스트 flush
        6. 트랜잭션 종료
        7. 영속성 컨텍스트 close(OSIV)
    3. 같은 엔티티 매니저라도 다른 트랜잭션이면 다른 영속성 컨텍스트 사용
        1. lock이 안걸려있으면 동시에 같은 거 접근 가능
        2. lock일 시 트랜잭션 단위로 수행됨 : 적용 → 적용 돼서 두번째거만 반영되게 보일 수도 있음
5. 상태
    1. 비영속 : 객체는 생성됐지만 영속성 컨텍스트에 관리 안됨. DB, JPA 관련 없을 때
    2. 영속 : 객체가 영속성 컨텍스트에 저장, 관리됨. 1차 캐시, 쓰기 지연, 변경 감지 대상
    3. 준영속 : 영속 상태였으나 분리됨 → 동작 안함 → 영속 변환
    4. 삭제 : 영속 상태였으나 분리됨 → 동작 안함 → 준영속 변환

영속 상태가 되면 JPA의 관리를 받게 되고 1차 캐시에 저장되고, 스냅샷됨
쿼리문 자체를 쓰기 지연 저장소에 저장
트랜잭션이 끝남과 동시에 모든 쓰기 지연 저장소에 있는 것들을 flush해서 한번에 적용함 -> 이때 스냅샷과 1차 캐시에 변경사항이 감지되면 쿼리 실행

준영속 -> 변경 감지를 막고 싶을 때 사용 DB에 적용 안됨
삭제 -> 진짜 DB에서 삭제하기 때문에 영속에서 제외 (어차피 데이터 없으니까)

#### 주제 7: 연관관계 매핑

엔티티 객체 간의 관계를 설정

왜 사용? 데이터 정합성과 설계 일관성을 유지

- entity를 이용해 객체로 만듦 → 이를 DB와 동일하게 연결

단방향 연관관계

- 한 엔티티에서만 다른 엔티티를 참조하는 관계
- 참조를 가진 쪽은 반대편에 접근 가능
- 반대편은 접근 불가능
- DB는 양쪽 JOIN이 가능하지만, 객체에서는 참조만 함
    - 양방향 모두 참조를 만들면 생성 수정 시 양쪽 참조를 항상 일치시켜야 함 > 복잡+ 규칙관리+
- 설계 단순화, 불필요한 의존성 삭제하기 위해 사용

양방향 연관관계

- 외래키를 관리하는 테이블이 주체임 > FK가 있는 테이블이 주체
- 연관된 엔티티를 양쪽에서 조회해 편의성 탐색 범위를 넓히기 위해
- 서비스에서 양쪽 참조가 모두 필요할 때 사용
- 초기에는 단방향 설정 → 양방향 추가

#### 주제 8: 프록시

JPA에서 지연 로딩을 사용할 때, 실제 엔티티 대신 먼저 생성되어 전달되는 가짜 객체

지연 로딩 ↔ 즉시 로딩 : 모든 객체를 즉시 조회 - 연관 엔티티를 lest outer join으로 한번에 조인 → null, 불필요 조인 추가되어 성능에 좋지 않음

- 기본 로딩 전략
    - ManyToOne
        - EARGER
    - OneToOne
        - EARGER
    - OneToMany
        - LAZY
    - ManyToMany
        - LAZY
