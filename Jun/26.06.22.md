#### 주제 2: - VPC
하나의 서버 안에서 가상 서버를 구분하기 위해서 사용되는 논리적으로 분리된 가상 네트워크

서버와 DB를 인터넷에 연결하지 않고 서비스 별 통신 경로와 접근 범위를 통제하기 위해 사용

구성 요소

1. IPv4 CIDR : /16 ~ /28
2. 서브넷 : IP 주소 범위를 더 작은 네트워크 단위로 나눔
    1. 퍼블릭 서브넷 : 인터넷 연결 가능. 입출입 가능 → 보안 떨어짐
    2. 프라이빗 서브넷 : 인터넷 연결 불가능. 입출입 불가능 → 보안 굿
    
    → 라우팅 테이블에 인터넷 게이트웨이 경로가 있냐 없냐로 구분
    
3. AZ(가용 영역) : 데이터센터의 위치
    1. 보통 서브넷을 나눌 때는 최소 한두개 이상의 AZ에 분배되도록 함 → 분산
4. 라우팅 테이블 : 네트워크 경로표
5. 인터넷 게이트웨이 : VPC랑 인터넷을 연결하는 출입구
    1. 퍼블릭 서브넷 라우팅 테이블에 0.0.0.0/0 → IG 0번 경로 추가
6. NAT Gateway : 프라이빗 서브넷에서 외부로 나갈 수 있게 하나 들어오는 건 막음

주의사항

1. 퍼블릭 IP 개당 과금이기 때문에 조심하기
2. 퍼블릭 IP 내부 서비스 할당 조심 ex) RDS, ElasticCache 등
3. ssh 열 때, 0.0.0.0/0 → pem 가지고 있으니 열어도 된다? 아님.
4. VPC 할당할 때 설계하기 → 서비스 구성 요소들을 어디에 둘 지(각 서브넷)
5. NAT Gateway 조심하기 → 상당히 비쌈

#### 주제 3: IAM
AWS 리소스에 대한 사용자 인증과 권한 관리 제공

IAM이 다루는 두가지 : 인증/인가

처음에 최소 권한으로 할당 후 필요할 때 확장하는 것이 안전함
 
- s3, ec2, cloudFront full

Group : 여러 IAM 유저에게 권한을 한번에 할당할 때 사용자를 묶음

- Developers, Operators, ReadOnlyAuditors, BillingAdmins

Policy : 어떤 작업을 허용할지 거부할 지 정함

    ```json
    // 특정 버킷에 파일 업로드만 허용
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": "s3:PutObject",
          "Resource": "arn:aws:s3:::my-service-image/image/*"
        }
      ]
    }
    ```

role : 특정 권한을 임시로 빌려서 사용할 수 있게 하는 권한 묶음

MFA : 멀티 팩터 인증 - 비밀번호 외 인증 수단들로 로그인할 때 사용하는 인증수단

#### 주제 4: 보안그룹
클라우드 환경에서 인스턴스에 대한 네트워크 트래픽을 제어하는 가상 방화벽

적용 범위 : 인스턴스 자체에 붙는 게 아닌, 네트워크 인터페이스에 연결함

사용 이유 : 무작정 노출하지 않기 위해서(필요대상에게 필요통신만제공)

특징 : 기본적으로 인바운드(밖에서 안으로) 규칙이 없으면 모든 트래픽을 허용하지 않음

- 상태 저장형 방화벽 : 한 번 허용된 요청의 응답은 반대 방향 규칙을 별도로 작성하지 않아도 자동으로 허용됨
- Allow는 가능 Deny 불가능 : 허용 규칙이 없으면 기본적으로 차단되어 명시적인 Deny이 필요없음
- 여러개 연결 가능

구성 : 타입, 프로토콜, 포트 범위, 인바운드 출발지, 아웃바운드 목적지, 규칙 목적 설명

보안 그룹 참조 : 보안 그룹을 인바운드, 아웃바운드로 설정 가능

#### 주제 5: Route53

사람이 읽는 도메인 주소 → 서버나 AWS 서비스의 주소로 연결하고 상황에 따라 어느 서비스에 보낼지 결정하는 DNS 서비스

사용 이유 : AWS랑 유기적으로 연결이 되지 않음

Hosted Zone

- 서브 도메인 사용하기
    - 프리픽스 사용하면 같은 ALB에 부담될 수 있음

Public Hosted Zone

- 인터넷 전체에서 조회할 수 있는 호스트 : 프론트 도메인
    - 공개 웹사이트
    - 공개 API
    - 외부 DNS 조회 가능

Private Hosted Zone

- VPN 내부에서 조회 가능
    - 외부 DNS 조회 불가능
    - VPC 안에서만 통신할 때 사용하는 도메인
    - IP가 생겼다 없어짐

NS 레코드

- Name Server Record : 해당 도메인의 DNS 설정 관리 주체

SOA

- 이 DNS 영역의 관리 기준 정보

DNS 레코드

- A : 도메인 → IPv4 주소
    - 고정된IP를 사용하는 경우에 사용
- CNAME : 도메인 → 다른 도메인
    - 한 도메인의 이름을 다른 도메인 이름에 연결
    - DNS 규칙 상 최상위 도메인에서 다른 도메인으로 이동시키는 건 불가능
- Alias : Route53 전용 AWS 리소스 연결 방식
- TXT : 메모장
    - 도메인 소유권 확인, 이메일 보안 정책
    - 텍스트 정보를 저장하는 레코드

라우팅 정책

1. Simple Routing : 하나의 도메인에 하나의 IP를 할당 전달
2. Weighted Routing : 여러 대상에 대해 비율대로 요청을 분산(가중치)
3. Failover Routing : 주 서버가 정상이면 주 서버로 전송, 비정상이면 부서버로 전송
4. Latency Routing : 네트워크 지연 시간을 기준으로 상대적으로 빠른 AWS 리전에 라우팅
5. Geolocation Routing : 사용자 IP 기반으로 가장 가까운 대륙에 있는 리전으로 라우팅

#### 주제 6: EC2
하이퍼바이저 : 물리 서버 한 대에 CPU, 메모리, 디스크, 네트워크 등의 자원을 여러개가 나눠서 사용하고 격리하는 소프트웨어 계층

가상화 : 하나의 물리 컴퓨터를 여러개의 독립된 가상 컴퓨터로 사용할 수 있도록 하는 기술

하이퍼바이저는 이용해 가상화 공간을 분리하고 CPU 자원 등을 분배한다. 각 가상화 공간에는 OS 가 존재함

    ```sql
    AWS 데이터센터
    └── 물리 서버
        └── 하이퍼바이저 (Nitro)
            └── VPC (가상 네트워크)
                └── EC2 (VM)
    ```

온프레미스보다 비싸지만 초기비용이 저렴함

핵심 요소

1. AMI : 서버 만드는 템플릿 - 운영체제 + 초기 소프트웨어 구성
    1. JDK 설치, 도커 설치 등등 … 미리 해두기
2. 인스턴스 타입 : CPU + 메모리 성능의 조합
    1. 어떤 타입 사용? CPU 점유가 평균 40%, RAM free가 평균 20% 쪽으로 부하테스트 하고 적용
3. EBS : EC2에 연결되는 “영구 블록 스토리지”
4. 네트워크 : EC2는 VPC 내 서브넷 안에 배치됨
5. 보안그룹
