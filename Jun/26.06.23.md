### 새로 배운 내용
#### 주제 1: EFS
스토리지 용량을 관리하지 않아도 자동으로 확장/축소되고 운영이 자동화된 서비스

EC2가 두대라면 서버 디스크에 파일을 저장했을 때 문제가 생길 수도 있음 → 공유 파일 시스템으로 해결(NFS)

AWS가 제공하는 NFS가 EFS

**마운트**

- 개념 : 외부 파일 시스템을 서버의 특정 디렉토리에 연결해서 로컬 폴더처럼 사용

다중 인스턴스를 사용하게 되면 EFS가 필요하게 됨

1. env 공유나 서버 디스크에 저장하는 파일
2. 용량을 예상할 수 없을 때
3. 여러 가용영역에서 공유할 때

**구성 요소**

1. 파일 시스템
2. Mount Target : EFS에 접속하기 위한 네트워크 진입점
    1. 각 AZ별로 있음
3. Security Group : VPC 내부에서 통신이 가능하게 설정하기

| 구분 | EBS | EFS | S3 |
| --- | --- | --- | --- |
| 저장 방식 | 블록 스토리지 | 파일 스토리지 | 객체 스토리지 |
| 접근 방식 | EC2 디스크처럼 연결 | 여러 서버에서 마운트 | API·SDK·HTTP로 접근 |
| 여러 EC2 동시 접근 | 일반적으로 제한적 | 가능 | 가능하지만 파일 시스템 방식 아님 |
| 대표 사용처 | EC2 운영체제, DB 디스크 | 공유 업로드 폴더, 공유 작업 파일 | 이미지, 동영상, 백업, 로그 |
| 파일 시스템 명령어 | 가능 | 가능 | 기본적으로 불가 |
| 확장 방식 | 용량 증설 작업 필요 | 자동 확장 | 자동 확장 |
| 성능 특성 | 낮은 지연 시간·디스크 I/O | 공유 파일 시스템 | 대규모 객체 저장·배포 |
| 비용 관점 | 프로비저닝 용량 기준 | 저장 사용량 + 요청/전송 고려 | 저장량 + 요청 + 전송 고려 |

**성능 비용 관점**

1. 성능 모드 : 일반 설정 → 모니터링 하면서 성능 확인
    1. General Purpose : 낮은 지연 시간 중심 → 보통 이거 사용
    2. Max I/O : 높은 병렬 처리 중심
2. Throughput Mode
    1. Elastic : 사용에 따라 처리량 자동 조절 → 보통 이거 사용
    2. Bursting : 파일 시스템 크기와 버스트 크레딧 특성 활용
    3. Provisioned : 필요한 처리량을 직접 지정
3. Lifecycle Management : 스토리지, 파일 시스템의 저장된 것 들 중 버려진 것 관리
    1. 오랫동안 접근하지 않는 파일은 항상 저렴한 스토리지로 이동

**실무 적용 시 고려사항**

1. EFS는 모든 파일 저장소로 사용하지 않음 : 비용이 상대적으로 비싸기 때문
2. DB로 사용하지 않음 : RDS나 EC2 내부 설치 추천. 네트워크 통신 때문에
3. NFS Port 2049 인터넷 열지 않기
4. 백업 설계 필요 : AWS 백업 정책7

**실습**

apt install -y nfs-common

which mount.nfs

NFS 클라이언트 사용 명령어 복붙 → **마운트** 하는 것

#### 주제 2: S3
이미지, 동영상, 백업 파일 등을 객체 단위로 저장하고 인터넷 또는 AWS 내부 서비스에서 접근할 수 있게하는 서비스

서버 디스크에 저장했을 때 문제점 : 서버가 많아지면 공유가 어려움

근데 EFS로 묶기도 조금 애매할 때 : ex) 이 이미지 파일은 외부 이미지에서도 볼 수 있어야 함 → EFS는 공유 네트워크 파일 스토리지로, VPC 내부에서만 사용해야함

객체 스토리지 : S3에 저장되는 실제 데이터 단위. 파일 + 메타 데이터 저장

구성 요소

1. 버킷 : S3 객체를 담는 최상위 저장 공간 (최상위 폴더)
    1. 이름 중복 설정 안됨(계정마다)
2. Object : 실제로 저장되는 데이터 단위
3. Key : 버킷 안에서 객체를 구분하는 고유한 이름

사용 이유

1. 제한 없이 확장 가능 : 디스크 증설 필요 없음
2. 서버가 바뀌어도 파일이 유지 : 새로운 EC2가 IAM 권한이 있으면 접근 가능해지고, 그대로 사용 가능
3. 내구성과 가용성이 좋음
    1. 내구성 : 저장한 데이터가 사라지지 않을 가능성
    2. 가용성 : 필요할 때 서비스를 사용할 수 있는 가능성

사용법

1. 백엔드가 업로드
    - 파일 처리(포맷팅, 검증) 필요 → S3는 파일을 저장하고 조회하는 데 목표이기 때문에 악성 코드 들어오는 걸 막기 위해 필요
2. Presigned URL 구조
    - 일정 시간동안 특정 s3 버킷, 객체에 접근할 수 있게 만든 서명된 URL
    - 특정 객체에 제한 시간동안 업/다운로드 권한 제공됨
    - 백엔드 서버가 큰 파일을 직접 받지 않아도 돼서 좋음 → 네트워크 자원을 덜 써서 저렴하게 운영 가능
    - 파일에 대한 검증, 포맷팅 등을 클라이언트에게 맡겨야 함

주요기능

1. 버전 관리(Versioning) : 같은 키로 파일을 다시 올려도 이전 버전을 보관함
2. Lifecycle Rule : 오래된 객체를 더 저렴한 스토리지로 옮기기 가능
3. 스토리지 클래스 설정(성능) : 접근 속도, 복구 속도에 따라서 필요한 스토리지 클래스를 사용

사용처

1. 사용자 이미지 처리
2. 정적 웹사이트 배포

#### 주제 3: RDS
클라우드에서 관계형 데이터베이스를 손쉽게 생성,운영, 확장할 수 있도록 제공하는 완전 관리형 서비스

둘 다 EBS 에 설치되는데, RDS를 쓰면 운영까지 자동으로 해줌

운영 환경에서 private subnet에 두기 → ssh, https 접속 불가

**구성 요소**

1. DB 엔진 : MySQL, MariaDB, Oracle …
2. DB 인스턴스 : 실제 DB 엔진이 실행되는 서버 자원
    1. 성능, CPU, 메모리 등 확인
3. Storage : DB raw + 인덱스, 트랜잭션 로그, 자동 백업본 등
   1. 스토리지 사이즈, IOPS, Throughput
   - IOPS : Input Output Operation Pr Second - 디스크가 1초동안 처리할 수 있는 작업 횟수
       - PK, 인덱스 기반으로 조회할 때 중요
       - Update가 많을 때 중요
       - 랜덤 액세스가 많을 때 중요
   - Throughput : 디스크가 1초에 얼마나 많은 양의 데이터를 읽고 쓰는지
       - 대량 조회, 생성이 발생하는 경우 중요
       - 리포트 생성 등
4. Endpoint : 다른 앱애 RDS에 접근할 수 있는 지점
5. 보안그룹 : 어떤 서버가 접근할 수 있는지 설정 (개발자 계정 …)

**기능**

- 자동 백업 : RDS가 주기적으로 수행하는 관리형 백업
    - 잘못됐을 때 백업을 수행함
    - 지정 백업 기간에만 유지되고, 그 기간 안에서만 특정 시점으로 복구 가능
- 스냅샷 : 사용자가 원하는 시점에 직접 만드는 DB 스냅샷
    - DB의 큰 변화 작업 전에 미리 현재 상태를 찍어야 함

**실무 구성 방법**

1. Multi AZ : DB가 죽으면 서버도 죽음 → 다른 AZ에 대기 DB 유지 → 고가용성 업
2. Read Replica : insert, update, delete 는 진짜 DB에, select는 다른 서버로 분산
    1. 보통 웹서비스가 읽기가 훨씬 많기 때문에!
        1. 수정 삭제 등록은 진짜 DB
        2. 목록 조회의 경우에는 Read Replica

- RDS의 백업 기능을 신뢰하지 말아라
- DB 연결 수 무한 금지 : 스레드 풀에 대한 옵티마이징 필요
