### 새로 배운 내용
#### 주제 1: 프록시
클라이언트와 서버 사이에서 중계자 역할을 하는 네트워크 장치

**포워드 프록시**

- 클라이언트 대신 외부 서버에 요청을 보내는 프록시
- 특정 사이트 접근 제한, 클라이언트 IP 숨기기 등
- 우회나 익명 목적으로 사용할 수 있음 → 보안

**리버스 프록시**
- 서버 대신 요청을 받아 먼저 처리하는 프록시
- 엔진엑스, 아팟치, API 게이트웨이 등
- 서버로 요청을 서빙해주는 것

**프록시 정리**

| 핵심 | 설명 |
| --- | --- |
| 한줄 정리 | 클라이언트와 서버 사이의 중계자 “역할” |
| 핵심 역할 | 요청과 응답을 대신 전달 |
| 주요 목적 | 보안, 최적화, 접근 제어, 캐싱, 로깅, 익명 |
| Forward Proxy | 클라이언트의 대리자 |
| Reverse Proxy | 서버의 대리자 |
| 실무 예시 | Nginx, HAProxy, Envoy, API Gateway, CDN |
| 주의점 | 병목, 장애점, 헤더 처리, 타임아웃, 로그 해석 |

프록시는 보안 장비? → 아니다. 요청과 응답의 중계자. 보안 기능을 수행하도록 함

포워드, 리버스 프록시는 같은 장비? → 기술적으로는 소프트웨어로 다 구현할 수 있음 예를 들어 엔진엑스에서 둘 다 구현 가능

리버스 프록시랑 로드 밸런서는 다른 것임 근데 보통 하나의 서버가 두개의 역할을 함

#### 주제 2: VPN
인터넷에 지나가는 노드를 지나갈 때 서버와 통신하는 내용을 한 번 더 감싸서 통신하기 위해 연결하는 것 → 공용 인터넷을 사설망처럼 사용 가능

사용 이유 : 공용 인터넷은 보안이 좋지 않을 수 있음

- 원격지를 보호하기 위해 내부망 사용
- 보안성 강화 - 공용망에서 익명성이 생김
- 떨어진 사설망을 하나의 사설망처럼 묶을 때 필요

구성 요소 : 클라이언트, 서버, 터널, 인증 정보, 암호화 키, 가상 IP, 라우팅 정책

포워드 프록시의 역할을 수행함

**VPN 의 동작 흐름**

```
1. VPN Client 실행 
2. VPN Server로 연결 요청
3. 사용자 인증
4. 암호화 키 협상
5. VPN 터널 생성
6. 가상 IP 할당
7. 라우팅 테이블 변경
8. 트래픽 암호화 후 터널로 전송
9. VPN Server가 복호화 후 목적지로 전달
10. 응답도 다시 암호화되어 Client로 반환
```

터널링 : 원본 패킷을 다른 프로토콜로 감싸서 보내는 방식

연결 방식

1. Full Tunnel : 모든 트래픽을 항상 VPN으로 전송
2. Split Tunnel : 사내망에 가야하는 트래픽만 VPN

#### 주제 3: NAT
사설IP ↔ 공인IP 주소를 변환하는 기술

IPv4의 한계 : 0.0.0.0 ~ 255.255.255.255 → 약 40억개 가능 But 현재 약 42억개의 공개 IP가 존재

NAT Table : 출발지(사설IP), NAT 변환(공인IP), 목적지(서버IP)

**Symmetric NAT**
- 매번 포트를 할당함. 외부 대상에 따라 IP, Port 새로 할당.
- 특징 : 매핑 독립성, 보안 강화, P2P 연결 제약(비대칭 통신 제한), 리소스 사용 증가

**Cone NAT**
- 특정 내부 IP와 포트에 대해 NAT 매핑 고정. 같은 사람이 요청하면 기존 포트 재사용.
- 특징 : 외부 IP에서 매핑 가능

**Full Cone NAT**
- 한번이라도 외부에 요청하면 내부 노드에게 포트를 할당해줌
- 특징 : 외부에서 자유롭게 내부 노드 요청 가능, 내부는 한번만 요청하면 외부와 자유롭게 통신
- 보안 : 허용 범위가 넓어 보안에 취약, 비인가 접근 가능성, NAT 장치가 외부 요청을 별도 필터링하지 않음

**Restricted Cone NAT** : 외부 IP 접근 차단

**Port Restricted Cone NAT** : 외부 IP + 포트 번호까지 매칭 조건으로 추가함

#### 주제 4: 네트워크 프로토콜

네트워크 상 데이터를 주고받는 규칙과 절차를 정의하는 체계

데이터 전송을 원활하고 안전하게

**전송 계층 프로토콜 종류**

- TCP : 신뢰성 있는 데이터 전송 보장 → 데이터 순차 처리
- UDP : 비연결성 프로토콜 → 빠른 전송 속도
  - TCP는 3way handshake 를 통해 서로 통신을 확인하고, 이후 패킷 전송 → 응답
  - UDP는 데이터만 보냄 유실되어도 무방한 스트리밍 등에 사용됨
- ICMP : 네트워크 상태 확인
- ARP : IP 주소 기반으로 MAC Address 찾음

**응용 계층 프로토콜 종류**

- HTTP, HTTPS : 하이퍼텍스트 전송
- FTP : 파일 전송 프로토콜
- SSH : 원격 접속
- SMTP : 이메일 전송을 위한 프로토콜, 이메일 서버 간 메세지 전달

#### 주제 5: 패킷
데이터를 이어진 스트림이 아닌 패킷으로 쪼개서 보내자! → 대기 없이 많은 장비에서 통신

문제 : 패킷이 복잡해짐 → TCP/IP 사용(계층화)

- 응용 계층 : HTTP 등
- 전송 계층 : 데이터를 신뢰성 있게 만듦. 세그먼트 단위로 나눠서 전송. 전송 헤더를 만듦
- 인터넷 계층 : IP 헤더 등을 적음
- 네트워크 인터페이스 계층 : 맥 주소같은 헤더를 만듦

#### 주제 6: CIDR
라우팅 시 트래픽 경로를 명확하게 지정하기 위해 네트워크 범위를 구분하는 주소 표준

네트워크와 호스트 정보를 구분하지 않으면 모든 라우터는 모든 경로에 대해 다 알고 있어야 함

→ 분리함으로서 네트워크 공간 먼저 찾고, 그 안의 호스트를 찾는 단계별 서칭

→ 분리하는 기준 : 서브넷 마스크( / 뒤에)

- 0은 네트워크, 255는 브로드캐스트용이기 때문에 주소 할당에 제외
