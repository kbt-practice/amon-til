### 새로 배운 내용
#### 주제 1: 공유 자원
여러 사용자나 프로세스, 스레드가 동시에 적븐하거나 사용하는 자원

공유자원 : 운영체제의 자원이 유한하기 때문에 동시에 작업할 수 있는 공간을 만듦 → 메모리, 파일, 데이터베이스, 네트워크 소켓, 전역 변수

→ 메모리 절약, 데이터 일관성 유지, 프로세스 간 협업을 위해 사용

→ BUT 동시에 공유자원을 수정하면 하나의 반영이 누락되는 일이 발생할 수 있고, 경쟁에 따라 결과가 달라질 수 있음

#### 주제 2: 임계 영역
동시 접근 시 문제가 발생되는 공유 자원을 사용하는 코드 구간

임계 영역의 분리 : 데이터 무결성 유지, 경합 조건 방지, 예측 가능한 프로그램 실행, 시스템 안정성 확보를 위해

→ 경쟁에 따라 결과가 달라지는 문제, 데이터 무결성이 깨지는 문제, 교착 상태 발생 → BUT 개발자가 알기 어려움

→ 동기화로 해결 : 앞사람이 끝날때까지 기다렸다가 입장하고 일 다 보면 나옴(**상호 배제, 진행**), 오래 기다리면 앞으로 꺼내주고(**유한대기**), 필요한 구간만 보호함

#### 주제 3: 동기화 방법
Lock : 공유 자원에 들어가기 전 잠금 획득하고 작업 후에 잠금 해제함

1. Mutex : 한번에 하나의 스레드만 임계 영역에 들어가도록 함
2. 세마포어 : 임계 영역에 들어갈 수 있는 스레드 제한

**주의할 점**

Atomic 연산 : 읽기, 수정, 저장을 하나로 보는 것

- 카운터 증가는 아토믹 사용
- CAS 연산 : 내가 읽은 값이 그대로일 때 차감

트랜잭션 : 내부적으로 lock, mvcc, isolation level

임계 영역 작게 유지 : 너무 넓어지면 성능 저하됨

**실무**

개발자 : 트랜잭션, 레디스 락, 메세지 큐, 커넥션 풀, 동시 요청 처리

인프라엔지니어 : 커널 내부 락, 파일 시스템, 데이터베이스, 쿠버네티스 등 …

주의 : 임계 영역은 작게, 락 해제 잊지 말기, 락은 순서를 맞춰야 함, DB에서 유니크/트랜잭션 사용 많이 하기

#### 주제4: 락
공유 자원 앞에 걸어두는 자물쇠 → 쓰기 시 보호

→ 데이터 무결성 유지, 경합 조건 방지, 원자성 보장, 동기화 제공

**동작** (획득 → 작업 → 해제)

| 단계 | 설명 |
| --- | --- |
| 1. 락 요청 | 스레드가 공유 자원에 접근하기 전에 락을 요청한다. |
| 2. 락 획득 | 락이 비어 있으면 해당 스레드가 락을 획득한다. |
| 3. 임계 영역 실행 | 락을 가진 스레드만 공유 자원을 사용한다. |
| 4. 다른 스레드 대기 | 다른 스레드는 락이 해제될 때까지 기다린다. |
| 5. 락 해제 | 작업이 끝난 스레드가 락을 해제한다. |
| 6. 다음 스레드 진입 | 대기 중인 다른 스레드가 락을 획득하고 작업한다. |

**종류**

1. 스핀 락 : 락이 풀렸나 계속 확인
    1. 컨텍스트 스위칭 비용 적음, 커널 내부 또는 짧은 임계 영역에 적합
    2. 기다리는 동안 CPU 계속 사용
    3. 백엔드 개발에서는 쓸 일이 거의 없음
2. 낙관적 락 : 충돌 거의 없다고 가정
    1. 작업 먼저 마지막에 충돌 여부 검사
3. 뮤텍스 : 한번에 하나의 스레드만 접근 가능
    1. 잠김 상태, 소유 스레드ㅡ 대기큐 정보 관리
    2. 핵심

    | 핵심 | 설명 |
    | --- | --- |
    | 상호 배제 | 한 번에 하나의 스레드만 임계 영역에 들어갈 수 있다. |
    | 소유권 | 락을 획득한 스레드가 “현재 소유자”가 된다. |
    | 해제 권한 | 일반적으로 락을 획득한 스레드만 락을 해제할 수 있다. |
3. 누가 다음 락을 획득할 지는 대기큐에 있는 것들 중 하나가 선택됨
4. 리드 라이트 락 : 읽기는 다중 허용, 쓰기는 단일 허용
    1. 구현 복잡함, 높은 동시성
